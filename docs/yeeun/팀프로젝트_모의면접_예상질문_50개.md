# 팀프로젝트 모의 면접 예상질문 50개와 답변

작성일: 2026-05-07

Harvest Slot 사용자 예약 웹 프로젝트를 기준으로, 팀프로젝트 발표와 모의 면접에서 나올 수 있는 질문과 답변을 정리한 문서다.

## 1. 프로젝트 이해도

### 1. Harvest Slot 프로젝트의 핵심 목표를 한 문장으로 설명해 주세요.

답변:
Harvest Slot은 고객이 농가가 확정한 수확 예정 슬롯을 보고 상품을 예약한 뒤, 수확과 선별 이후 직배송 흐름까지 확인할 수 있는 예약 기반 농산물 주문 서비스입니다. 일반 쇼핑몰처럼 즉시 재고를 판매하는 방식이 아니라, 수확 예정 범위와 예약 가능 수량을 기준으로 주문이 이어지는 것이 핵심입니다.

### 2. 이 프로젝트에서 사용자 웹이 담당한 범위는 어디까지인가요?

답변:
제가 담당한 사용자 웹 범위는 고객이 실제로 보는 예약 웹 화면과 화면 기능 구현입니다. 홈, 상품 목록, 상품 상세, 수확 슬롯 선택, 로컬 예약함, 예약 확인, 주문서 작성, 결제 확인, 주문 완료, 주문 상세, 배송 확인, 반품 신청 흐름을 구현 대상으로 봤습니다. 백엔드 API 자체 구현이나 관리자 앱, 머신러닝 모델 구현은 범위 밖이지만, 화면에서 연결해야 하는 데이터와 상태는 이해하고 반영했습니다.

### 3. 관리자 앱, 백엔드, 머신러닝 담당 영역과 사용자 웹 담당 영역은 어떻게 구분했나요?

답변:
사용자 웹은 고객이 이해하고 조작하는 화면을 담당하고, 관리자 앱은 농가나 운영자가 상품, 수확 슬롯, 주문 상태를 관리하는 영역으로 분리했습니다. 백엔드는 예약 생성, 주문 생성, 결제 승인, 상태 변경 같은 서버 데이터 처리를 담당합니다. 머신러닝은 수확량 예측이나 운영 판단을 돕는 참고 데이터를 만드는 영역이고, 고객 화면에는 예측값 자체를 직접 노출하지 않도록 구분했습니다.

### 4. 사용자가 이 서비스를 이용하면 기존 구매 방식보다 어떤 점이 좋아지나요?

답변:
고객은 수확 예정 범위와 예약 가능 수량을 보고 미리 주문할 수 있어서, 상품이 언제 준비되는지 더 명확하게 알 수 있습니다. 또 수확 직배송이라는 흐름을 화면에서 계속 확인할 수 있어 신선도와 배송 과정에 대한 신뢰를 줄 수 있습니다. 특히 주문 후에도 농가 확인, 선별, 배송 중 같은 상태를 단계별로 보여주기 때문에 막연히 기다리는 느낌을 줄일 수 있습니다.

### 5. 수확 슬롯 기반 예약 방식이 일반 상품 주문 방식과 다른 점은 무엇인가요?

답변:
일반 상품 주문은 현재 재고를 바로 구매하는 구조에 가깝지만, 수확 슬롯 기반 예약은 농가가 확정한 수확 예정 기간과 예약 가능 수량을 기준으로 주문합니다. 그래서 상품 상세에서 슬롯을 선택하고, 예약 확인을 거친 뒤 서버 예약 생성과 주문 생성으로 이어집니다. 또한 기상과 생육 상황에 따라 일정이 조정될 수 있다는 안내가 중요합니다.

### 6. 고객 화면에서 ML 예측값을 직접 보여주지 않은 이유는 무엇인가요?

답변:
ML 예측값은 운영자나 농가가 수확 슬롯을 판단하기 위한 참고 데이터에 가깝습니다. 고객에게 예측값을 그대로 보여주면 확정 정보처럼 오해할 수 있고, 실제 수확 일정이나 수량과 다를 때 신뢰 문제가 생길 수 있습니다. 그래서 고객 화면에는 농가가 확정한 수확 예정 범위와 예약 가능 수량만 보여주는 방향으로 설계했습니다.

### 7. 프로젝트에서 가장 중요하게 생각한 사용자 흐름은 무엇인가요?

답변:
가장 중요한 흐름은 홈에서 상품을 보고, 상품 상세에서 수확 슬롯을 선택하고, 로컬 예약함과 예약 확인을 거쳐 주문과 결제까지 이어지는 흐름입니다. 이 흐름이 끊기면 서비스의 핵심인 예약 주문이 성립하지 않습니다. 이후 주문 상세, 배송 확인, 반품 신청도 중요하지만, 우선 고객이 예약을 이해하고 완료할 수 있는 흐름을 안정적으로 만드는 것이 핵심이라고 봤습니다.

### 8. 기획서와 실제 구현 사이에 차이가 생겼을 때 어떤 기준으로 판단했나요?

답변:
첫 번째 기준은 사용자 예약, 주문, 배송 확인 흐름을 더 명확하게 만드는지였습니다. 두 번째는 기존 API나 작은 응답 필드 추가 정도로 구현 가능한지였습니다. 세 번째는 관리자 앱이나 ML, 백엔드 담당 범위를 침범하지 않는지였습니다. 예를 들어 기본 배송지 선택이나 주문 상태 타임라인은 사용자 흐름을 명확하게 하면서도 사용자 웹 범위 안에서 설명 가능한 기능입니다.

### 9. 사용자 웹에서 가장 중요한 UX 키워드는 무엇이라고 생각하나요?

답변:
신선함, 정직한 농가, 예약 안정성, 수확 직배송, 명확한 상태 안내가 중요하다고 봤습니다. 특히 예약 서비스이기 때문에 고객이 지금 무엇을 예약하고 있고, 결제 후 어떤 단계에 있는지 계속 이해할 수 있어야 합니다. 그래서 상태 배지, 안내 박스, 주문 상태 타임라인 같은 요소를 중요하게 생각했습니다.

### 10. 이 프로젝트를 실제 서비스로 확장한다면 가장 먼저 보완할 부분은 무엇인가요?

답변:
가장 먼저 실제 API 연결과 에러 처리, 인증 흐름을 보완해야 한다고 생각합니다. 현재 더미 데이터 기반 화면은 흐름을 설명하기에는 좋지만 실제 서비스에서는 슬롯 품절, 예약 만료, 결제 실패, 배송지 오류 같은 상황을 안정적으로 처리해야 합니다. 그 다음으로는 접근성, 모바일 반응형 완성도, 주문 상태 알림 기능을 보완하고 싶습니다.

## 2. 사용자 흐름과 화면 설계

### 11. 홈에서 주문 완료까지의 전체 사용자 흐름을 설명해 주세요.

답변:
사용자는 홈에서 추천 상품을 확인하고 상품 목록으로 이동합니다. 상품 상세에서 상품 정보와 수확 예정 범위를 확인한 뒤 수확 슬롯과 박스 수량을 선택합니다. 선택한 항목은 로컬 예약함에 담기고, 예약 확인 화면에서 수량과 가격을 확인한 뒤 서버 예약을 생성합니다. 이후 주문서 작성에서 배송 정보를 입력하고, 결제 확인을 거쳐 Mock 결제가 승인되면 주문 완료 화면으로 이동합니다.

### 12. 상품 상세 화면에서 사용자가 반드시 확인해야 하는 정보는 무엇인가요?

답변:
상품명, 농가명, 품종, 포장 단위, 판매가, 수확 예정 범위, 예약 가능 수량, 선택 가능한 슬롯이 중요합니다. 특히 이 서비스는 수확 일정 기반이므로 "농가가 확정한 수확 예정 범위이며 기상과 생육 상황에 따라 일정이 조정될 수 있다"는 안내가 필요합니다. 고객이 구매 시점에 확정 정보와 변동 가능 정보를 구분할 수 있어야 합니다.

### 13. 수확 일정 변동 가능성 안내 문구를 넣은 이유는 무엇인가요?

답변:
농산물은 기상과 생육 상황에 영향을 많이 받기 때문에 수확 예정일이 일반 배송 예정일처럼 고정된 날짜라고 보기 어렵습니다. 안내 문구를 넣으면 고객이 예약의 특성을 이해하고 주문할 수 있습니다. 이는 고객 신뢰를 높이고, 추후 일정 조정이 발생했을 때 혼란을 줄이는 역할을 합니다.

### 14. 로컬 예약함을 둔 이유는 무엇인가요?

답변:
로컬 예약함은 서버 예약을 만들기 전 고객이 선택한 슬롯과 수량을 임시로 확인하고 수정하는 단계입니다. 고객이 상품을 담았다는 이유만으로 서버에 예약 기록을 만들면, 실제 주문 의사가 없는 데이터가 서버에 쌓일 수 있습니다. 그래서 브라우저 저장소와 Flutter 상태로 먼저 관리하고, 예약 확인 후에 서버 예약 생성 API를 호출하는 구조가 적절하다고 봤습니다.

### 15. 고객이 kg를 직접 입력하지 않고 박스 수량을 선택하게 한 이유는 무엇인가요?

답변:
상품은 포장 단위가 정해져 있기 때문에 고객이 kg를 직접 입력하면 실제 판매 단위와 맞지 않을 수 있습니다. 예를 들어 5kg 박스 상품이면 수량은 박스 개수로 선택하고, 총 중량은 `package_count * package_unit_kg`로 계산하는 것이 더 명확합니다. 이렇게 하면 고객 입력 오류를 줄이고, 서버 검증도 단순해집니다.

### 16. 예약 확인 화면과 주문서 작성 화면의 역할 차이는 무엇인가요?

답변:
예약 확인 화면은 선택한 수확 슬롯, 박스 수량, 총 중량, 예상 가격을 확인하고 서버 예약을 생성하기 전 검증하는 단계입니다. 주문서 작성 화면은 이미 생성된 예약을 기반으로 받는 사람, 연락처, 배송지, 배송 요청사항 같은 주문 정보를 입력하는 단계입니다. 즉 예약 확인은 상품과 수확 조건 확인, 주문서 작성은 배송과 주문 정보 입력에 초점이 있습니다.

### 17. 결제 후 바로 배송 완료처럼 보이지 않게 문구를 정리한 이유는 무엇인가요?

답변:
이 서비스는 결제 직후 상품이 바로 배송되는 구조가 아니라, 농가 확인과 선별 과정을 거친 뒤 배송됩니다. 그래서 결제 완료 후 화면에서 "배송 완료"처럼 보이면 고객이 실제 진행 상태를 오해할 수 있습니다. 결제 후에는 "농가 확인 중" 또는 "수확 가능 수량 확인 중"처럼 현재 단계에 맞는 문구를 보여주는 것이 더 정확합니다.

### 18. 주문 상세 화면에서 상태 타임라인을 보여주는 이유는 무엇인가요?

답변:
예약 기반 주문은 결제 후에도 농가 확인, 선별, 발송 준비, 배송 중, 배송 완료처럼 여러 단계가 있습니다. 상태 타임라인을 보여주면 고객이 현재 주문이 어디까지 진행됐는지 한눈에 이해할 수 있습니다. 단순 상태 텍스트보다 전체 흐름 속 현재 위치를 보여주는 것이 고객 불안을 줄이는 데 도움이 됩니다.

### 19. 반품 신청 버튼을 배송 완료 상태에서만 노출한 이유는 무엇인가요?

답변:
반품은 고객이 상품을 받은 뒤에 신청하는 흐름이기 때문에 배송 전 단계에서는 노출하지 않는 것이 맞습니다. 배송 중이거나 농가 확인 중인 상태에서 반품 버튼이 보이면 기능 의미가 혼동될 수 있습니다. 따라서 주문 상태가 `DELIVERED`, 즉 고객 표시로는 "배송 완료"일 때만 반품 신청이 가능하도록 설계했습니다.

### 20. 마이페이지, 배송지 관리, 회원 정보 화면을 추가한 이유는 무엇인가요?

답변:
예약 주문 흐름에서는 주문 내역 확인과 배송지 관리가 중요합니다. 고객은 주문 후 상태를 다시 확인해야 하고, 반복 구매 시 기본 배송지가 있으면 주문서 작성이 더 편해집니다. 회원 정보 화면은 로그인한 사용자가 본인 정보를 확인하고 수정할 수 있는 기본 기능으로, 사용자 웹의 완성도를 높이는 보조 흐름입니다.

## 3. Flutter와 MVVM 구조

### 21. 이 프로젝트에서 MVVM 구조를 선택한 이유는 무엇인가요?

답변:
화면 표시, 상태 관리, 데이터 호출 책임을 분리하기 위해 MVVM 구조를 선택했습니다. Flutter 화면이 많아질수록 View 안에 검증 로직이나 API 호출이 섞이면 유지보수가 어려워집니다. MVVM으로 나누면 View는 화면과 입력 전달에 집중하고, ViewModel은 상태와 검증, Repository는 API와 데이터 변환을 맡을 수 있습니다.

### 22. View, ViewModel, Repository, Model의 역할을 각각 설명해 주세요.

답변:
View는 사용자에게 화면을 보여주고 버튼 클릭이나 입력을 ViewModel에 전달합니다. ViewModel은 로딩, 에러, 선택값, 검증 결과 같은 화면 상태를 관리하고 Repository를 호출합니다. Repository는 API 호출과 응답 변환을 담당합니다. Model은 API 요청, 응답, 화면에서 사용할 데이터 구조를 표현합니다.

### 23. View에서 API를 직접 호출하지 않도록 한 이유는 무엇인가요?

답변:
View에서 API를 직접 호출하면 화면 코드가 데이터 처리 방식에 강하게 묶입니다. 그러면 같은 데이터를 다른 화면에서 재사용하기 어렵고, 테스트하기도 어려워집니다. ViewModel과 Repository를 거치면 화면은 상태만 구독하고, API 변경이 생겼을 때 Repository나 Model 쪽에서 대응하기 쉬워집니다.

### 24. ViewModel에는 어떤 상태와 로직을 두었나요?

답변:
ViewModel에는 화면에 필요한 로딩 상태, 에러 메시지, 선택된 상품이나 슬롯, 입력값 검증, 버튼 활성화 조건 같은 로직을 둡니다. 예를 들어 예약 확인에서는 선택한 박스 수량, 총 중량, 가격, 예약 가능 여부를 관리할 수 있습니다. 주문서 작성에서는 배송지 입력값과 필수값 검증을 ViewModel이 담당하는 구조가 적절합니다.

### 25. Repository 계층을 별도로 둔 장점은 무엇인가요?

답변:
Repository를 두면 API 호출 방식과 화면 로직을 분리할 수 있습니다. 나중에 더미 데이터에서 실제 API로 바꿀 때 View를 크게 수정하지 않고 Repository 구현을 바꾸면 됩니다. 또한 API 응답 구조가 바뀌어도 Repository에서 화면에 필요한 Model로 변환해주면 ViewModel의 변경 범위를 줄일 수 있습니다.

### 26. 공통 위젯을 분리하면 어떤 장점이 있나요?

답변:
공통 위젯을 분리하면 화면 전체의 디자인 일관성을 유지할 수 있고, 반복되는 코드를 줄일 수 있습니다. 예를 들어 상태 배지를 여러 화면에서 각각 만들면 색상이나 문구가 달라질 수 있습니다. `StatusBadge`, `NoticeBox`, `PriceText` 같은 위젯으로 분리하면 수정이 필요할 때 한 곳에서 관리할 수 있습니다.

### 27. ProductCard, HarvestSlotCard, StatusBadge 같은 공통 컴포넌트는 어떤 기준으로 만들었나요?

답변:
여러 화면에서 반복되고, 의미가 명확한 UI 단위를 기준으로 만들었습니다. `ProductCard`는 상품 목록과 추천 상품에서 반복되고, `HarvestSlotCard`는 수확 슬롯 선택에서 반복됩니다. `StatusBadge`는 주문 상태나 배송 상태를 고객 친화적인 문구로 보여주는 역할이 있어서 공통화하기 좋습니다.

### 28. Flutter Web에서 반응형 화면을 만들 때 가장 신경 쓴 부분은 무엇인가요?

답변:
모바일과 데스크톱에서 정보가 잘리지 않고 읽히는 것을 가장 신경 썼습니다. 특히 카드, 상단 메뉴, 버튼 문구, 가격 정보처럼 폭이 좁아졌을 때 overflow가 발생하기 쉬운 요소를 조정해야 합니다. 화면 크기에 따라 그리드 개수, 여백, 카드 높이를 다르게 하고, 긴 문구는 줄바꿈될 수 있게 구성하는 것이 중요합니다.

### 29. 모바일 화면에서 카드 overflow를 줄이기 위해 어떤 접근을 했나요?

답변:
모바일에서는 카드 안의 이미지 높이와 여백을 줄이고, 텍스트가 너무 길면 줄바꿈되도록 구성하는 방식이 필요합니다. 상품 카드의 정보 우선순위를 정해서 핵심 정보인 상품명, 가격, 수확 범위, 예약 가능 상태가 먼저 보이게 합니다. 필요하면 데스크톱과 모바일에서 카드 레이아웃을 다르게 가져가는 것도 방법입니다.

### 30. AppBar를 공통화했을 때 얻은 이점은 무엇인가요?

답변:
여러 화면에서 브랜드명, 홈 이동, 주요 메뉴가 같은 방식으로 동작하게 만들 수 있습니다. AppBar를 화면마다 따로 만들면 메뉴 문구나 이동 경로가 달라질 수 있고, 수정이 필요할 때 누락될 위험이 있습니다. 공통 AppBar를 사용하면 사용자 경험이 일관되고 유지보수도 쉬워집니다.

## 4. 상태 관리와 데이터 처리

### 31. 로컬 예약함 데이터는 서버 예약 데이터와 어떻게 다르게 관리되나요?

답변:
로컬 예약함은 고객이 선택한 상품과 슬롯을 임시 저장하는 프론트 상태입니다. 서버 예약 데이터는 고객이 예약 확인을 완료하고 서버 예약 생성 API가 성공한 뒤 만들어지는 공식 기록입니다. 따라서 로컬 예약함 단계에서는 서버에 예약이 확정되지 않았고, 최종 검증도 서버 예약 생성 시점에 이루어집니다.

### 32. 예약 생성 전까지 서버에 예약 기록을 만들지 않는 이유는 무엇인가요?

답변:
고객이 단순히 상품을 담아보는 행동까지 서버 예약으로 만들면 불필요한 예약 데이터가 많이 생길 수 있습니다. 실제 주문 의사가 확인되지 않은 상태에서 수량을 점유하는 것도 문제가 될 수 있습니다. 그래서 고객이 예약 확인을 거쳐 진행하기로 결정한 시점에 서버 예약을 생성하는 것이 더 안정적입니다.

### 33. `package_count * package_unit_kg` 계산 규칙은 어디에 반영되어야 하나요?

답변:
프론트에서는 고객에게 총 중량을 보여주기 위해 계산하고, 서버에서는 최종 검증을 위해 다시 계산해야 합니다. 프론트 계산은 안내와 UX를 위한 것이고, 서버 계산은 신뢰 가능한 최종 데이터 처리를 위한 것입니다. 고객이 kg를 직접 입력하지 않도록 하고 박스 수량만 전달하면 계산 규칙이 명확해집니다.

### 34. 최종 수량과 가격 검증을 서버에서 처리해야 하는 이유는 무엇인가요?

답변:
프론트 데이터는 사용자가 조작할 수 있기 때문에 최종 검증 기준으로 삼을 수 없습니다. 예약 가능 수량, 슬롯 상태, 가격은 결제 직전에도 바뀔 수 있으므로 서버에서 최신 데이터 기준으로 확인해야 합니다. 서버 검증을 통해 품절, 가격 변경, 예약 만료 같은 문제를 막을 수 있습니다.

### 35. 주문 상태 코드를 고객에게 그대로 보여주지 않은 이유는 무엇인가요?

답변:
`PROCUREMENT_REQUESTED` 같은 서버 상태 코드는 개발자와 시스템이 이해하기 위한 값입니다. 고객에게 그대로 보여주면 의미를 이해하기 어렵고 화면이 딱딱해집니다. 그래서 고객 화면에서는 "농가 확인 중", "선별 중", "배송 중"처럼 이해하기 쉬운 한글 문구로 변환해서 보여줍니다.

### 36. `PROCUREMENT_REQUESTED` 상태를 고객 화면에서는 어떻게 표현하나요?

답변:
`PROCUREMENT_REQUESTED`는 고객 화면에서 "농가 확인 중"으로 표현합니다. 결제는 완료됐지만 농가가 수확 가능 수량과 주문 가능 여부를 확인하는 단계이기 때문입니다. 이 문구를 사용하면 고객이 결제 이후에도 아직 농가 확인 과정이 남아 있다는 점을 자연스럽게 이해할 수 있습니다.

### 37. 배송 상태와 주문 상태를 화면에서 어떻게 구분해 보여줄 수 있나요?

답변:
주문 상태는 결제, 농가 확인, 선별, 발송 준비 같은 주문 처리 전체 단계를 나타내고, 배송 상태는 실제 택배나 배송 진행 상태에 더 가깝습니다. 주문 상세 화면에서는 전체 진행 타임라인을 보여주고, 배송 정보 영역에는 택배사, 송장번호, 배송 중 또는 배송 완료 같은 정보를 분리해 보여줄 수 있습니다. 이렇게 나누면 고객이 주문 처리와 물류 진행을 구분해서 이해할 수 있습니다.

### 38. 더미 데이터로 구현한 화면을 실제 API와 연결할 때 가장 조심해야 할 점은 무엇인가요?

답변:
더미 데이터는 항상 정상 케이스만 담기 쉽기 때문에 실제 API 연결 시 빈 값, 에러, 로딩, 권한 만료, 상태 변경을 반드시 고려해야 합니다. 또한 화면에서 사용하는 필드명과 API 응답 필드명이 다를 수 있으므로 Model과 Repository에서 변환을 명확히 해야 합니다. 고객 화면에는 API 경로나 DB 테이블명 같은 개발자용 정보가 노출되지 않도록 주의해야 합니다.

### 39. 로그인하지 않은 사용자가 마이페이지에 접근하면 어떻게 처리해야 하나요?

답변:
비로그인 사용자가 마이페이지에 접근하면 로그인이 필요하다는 안내를 보여주고 로그인 화면으로 이동시키는 것이 적절합니다. 이때 사용자가 원래 가려던 경로를 저장해두면 로그인 후 다시 해당 화면으로 보내줄 수 있습니다. 단순히 빈 화면을 보여주거나 에러 상태만 보여주는 것은 사용자 경험이 좋지 않습니다.

### 40. 기본 배송지 선택 기능은 어떤 데이터가 필요하다고 생각하나요?

답변:
배송지 ID, 받는 사람 이름, 전화번호, 우편번호, 주소, 상세주소, 배송 요청사항, 기본 배송지 여부가 필요합니다. 여러 배송지 중 하나만 기본 배송지로 설정되어야 하므로 `is_default` 같은 필드가 필요합니다. 주문서 작성 화면에서는 기본 배송지를 먼저 보여주고, 사용자가 필요하면 다른 배송지를 선택하거나 수정할 수 있게 하는 것이 좋습니다.

## 5. API 협업과 백엔드 연결

### 41. 사용자 웹에서 백엔드와 연결해야 하는 주요 API를 설명해 주세요.

답변:
인증 영역에서는 회원가입, 이메일 인증, 로그인, 내 정보 조회 API가 필요합니다. 상품 영역에서는 추천 상품, 상품 목록, 상품 상세, 수확 슬롯 조회 API가 필요합니다. 예약과 주문 영역에서는 예약 미리보기, 예약 생성, 주문 생성, 결제 승인, 내 주문 목록, 주문 상세, 배송 정보, 반품 신청 API가 필요합니다. 기본 경로는 `/api/v1` 기준으로 정리했습니다.

### 42. 인증이 필요한 API에는 어떤 헤더가 필요하나요?

답변:
인증이 필요한 API에는 `Authorization: Bearer <access_token>` 헤더가 필요합니다. 로그인 성공 후 받은 access token을 저장하고, 내 정보 조회나 내 주문 목록처럼 개인 정보가 필요한 요청에 포함해야 합니다. 토큰이 없거나 만료된 경우에는 로그인 화면으로 이동하거나 재로그인을 안내해야 합니다.

### 43. 상품 목록과 상품 상세 화면에서 필요한 API 응답 필드는 무엇인가요?

답변:
상품 목록에서는 상품 ID, 상품명, 농가명, 대표 이미지, 가격, 포장 단위, 예약 가능 상태, 수확 예정 범위, 잔여 수량 같은 필드가 필요합니다. 상품 상세에서는 여기에 상품 설명, 품종, 상세 이미지, 슬롯 목록, 슬롯별 예약 가능 수량과 가격이 추가로 필요합니다. 고객에게 ML 예측값이 아니라 농가가 확정한 정보가 내려와야 합니다.

### 44. 예약 미리보기 API와 예약 생성 API의 차이는 무엇인가요?

답변:
예약 미리보기 API는 고객이 선택한 슬롯과 수량이 현재 기준으로 가능한지, 예상 금액이 얼마인지 확인하는 단계입니다. 예약 생성 API는 고객이 확인 후 실제 서버 예약 기록을 만드는 단계입니다. 미리보기는 검토 성격이 강하고, 예약 생성은 서버에 공식 예약 데이터가 생기는 차이가 있습니다.

### 45. 주문 생성 API에는 어떤 사용자 입력값이 필요하나요?

답변:
주문 생성 API에는 예약 ID와 배송 정보가 필요합니다. 배송 정보에는 받는 사람 이름, 연락처, 우편번호, 주소, 상세주소, 배송 요청사항이 포함될 수 있습니다. 서버는 예약이 유효한지 확인한 뒤 주문을 생성하고, 이후 결제 대기 상태로 이어지게 처리해야 합니다.

### 46. Mock 결제 승인 API가 성공하면 주문 상태는 어떻게 바뀌어야 하나요?

답변:
Mock 결제 승인 API가 성공하면 결제는 완료되고 주문은 농가 확인 단계로 넘어가야 합니다. 고객 화면에서는 결제 직후 "농가 확인 중"으로 보여주는 것이 자연스럽습니다. 서버 상태로는 결제 완료를 거쳐 `PROCUREMENT_REQUESTED` 흐름으로 이동하는 구조가 적절합니다.

### 47. 주문 상세 화면에서 배송 정보 API를 별도로 호출하는 이유는 무엇인가요?

답변:
주문 정보와 배송 정보는 갱신 주기와 책임이 다를 수 있습니다. 주문 상세는 상품, 결제 금액, 주문 상태를 보여주고, 배송 정보는 택배사, 송장번호, 배송 상태처럼 물류 관련 정보를 보여줍니다. 배송 정보가 아직 없을 수도 있으므로 별도 API로 분리하면 화면에서 "발송 준비 중" 같은 상태를 자연스럽게 처리할 수 있습니다.

### 48. 반품 신청 API에 전달해야 하는 값은 무엇인가요?

답변:
반품 신청 API에는 주문 ID, 반품 사유 코드, 상세 사유, 증빙 이미지, 요청 금액 같은 값이 필요합니다. 실제 서비스에서는 반품 가능한 주문 상태인지도 서버에서 다시 확인해야 합니다. 프론트에서는 배송 완료 상태에서만 버튼을 보여주더라도, 서버 검증은 반드시 필요합니다.

### 49. 프론트에서 백엔드 담당자에게 API 연결 지점을 문서화한 이유는 무엇인가요?

답변:
화면에서 필요한 데이터와 API 응답 필드를 명확히 공유하기 위해서입니다. 백엔드가 어떤 필드를 내려줘야 프론트 화면이 완성되는지 문서화하면 협업 중 누락을 줄일 수 있습니다. 또한 더미 데이터로 만든 화면을 실제 API로 전환할 때 어떤 Repository와 Model을 연결해야 하는지 기준이 됩니다.

### 50. API 응답 구조가 기획과 다를 때 프론트에서는 어떻게 대응해야 하나요?

답변:
먼저 화면에 꼭 필요한 정보가 응답에 있는지 확인해야 합니다. 필드명이 다르거나 구조만 다른 경우에는 Model이나 Repository에서 변환해서 대응할 수 있습니다. 하지만 필요한 데이터가 아예 없거나 의미가 다르면 백엔드 담당자와 API 스펙을 다시 조율해야 합니다. 이때 고객 화면에 필요한 문구와 상태 기준을 함께 설명하면 협업이 쉬워집니다.

## 추가 연습 방법

- 각 질문에 대해 30초 답변과 1분 답변을 따로 준비한다.
- 본인이 직접 구현한 화면은 코드 파일 위치와 함께 설명할 수 있게 정리한다.
- 면접 답변은 "왜 그렇게 했는지", "어떤 문제가 있었는지", "어떻게 개선할 수 있는지" 순서로 말하면 좋다.
- 답변을 외우기보다 핵심 키워드만 기억하고 본인 말투로 바꿔 말하는 연습을 한다.
