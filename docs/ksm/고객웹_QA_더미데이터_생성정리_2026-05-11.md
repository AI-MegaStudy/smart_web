# 고객 웹 QA 더미 데이터 생성 정리

작성일: 2026-05-11

## 1. 목적

고객 Web 화면 QA를 위해 예약, 주문, 결제, 배송, 반품 데이터가 서로 관계와 날짜 순서에 맞게 이어지는 더미 데이터를 생성했다.

핵심 조건:

- 예약이 있어야 주문이 있다.
- 주문이 있어야 결제가 있다.
- 결제 완료 후 배송 데이터가 있다.
- 배송 완료 후에만 반품 데이터가 있다.
- 환불 완료 데이터는 반품 요청과 승인된 결제 데이터를 함께 가진다.
- 기존 데이터는 삭제하지 않고 `QA-` prefix 데이터만 추가한다.

## 2. 생성 스크립트

```text
scripts/seed_customer_flow_qa_data.py
```

실행:

```powershell
python scripts\seed_customer_flow_qa_data.py
```

특징:

- 같은 스크립트를 여러 번 실행해도 `QA-RSV-*`, `QA-ORD-*`, `QA-RET-*`, `qa-payment-*` 기준으로 중복 생성되지 않게 구성했다.
- `Base.metadata.create_all(bind=engine)`를 호출해 누락된 테이블이 있으면 생성한다.
- DB 연결은 프로젝트 루트 `.env` 기준 백엔드 설정을 사용한다.

## 4. 생성 데이터 분포

각 고객 계정별로 아래 상태 데이터를 1건씩 생성했다.

```text
예약만 생성
주문 생성
결제 대기
결제 완료
배송 중
배송 완료
반품 신청
환불 완료
```

현재 생성 결과:

```text
QA 계정: 4개
예약: 24건
주문: 21건
결제: 18건
배송: 12건
반품: 6건
환불: 3건
```

주문이 없는 결제, 결제 없는 배송, 배송 완료 전 반품 데이터는 생성하지 않았다.

고객 화면에 보이는 대표 상품/농장명:

```text
문경 햇살 농장
충주 햇살 부사 사과
충주 햇살 신고 사과
```

표시값 갱신 스크립트:

```powershell
python scripts\refresh_customer_flow_display_data.py
```

신고 사과 혼합 스크립트:

```powershell
python scripts\mix_shingo_product_flow_data.py
```

부사/신고 상품 구성:

```text
충주 햇살 부사 사과: 5kg, 32,000원
충주 햇살 신고 사과: 5kg, 34,000원
```

고객 흐름 데이터 중 일부 예약/주문/결제/배송/반품은 신고 사과 슬롯으로 연결되도록 조정했다. 금액, 중량, 결제 금액, 반품/환불 금액은 연결된 상품 가격 기준으로 다시 계산했다.

## 5. 고객 Web QA 기준

고객 계정으로 로그인해서 아래 화면 흐름을 확인한다.

```text
예약 목록
주문 목록
주문 상세
결제 상태
배송 상태
반품 신청/반품 내역
```

고객 화면에는 `QA`, `더미`, `DB`, `API`, `Mock`, `ML` 같은 개발자용 표현이 직접 노출되지 않아야 한다.

## 7. 2026-05-11 고객웹 시연 마감 점검

고객웹 시연 전 화면 문구와 더미 데이터 노출값을 다시 점검했다.

반영 내용:

- 홈 품종 소개와 홈 배지에서 `양광` 문구를 `신고` 기준으로 변경했다.
- 로컬 더미 상품명도 `양광 사과` 대신 `신고 사과`로 정리했다.
- 주문 상품명 보정 로직에서 `신고 배`가 표시되지 않도록 `신고 사과`로 변경했다.
- 결제 대기 주문의 결제 정보에서 `0원` 승인금액 대신 결제 예정 금액이 보이도록 정리했다.
- 반품 요청 화면에서 증빙 이미지를 여러 장 선택할 수 있도록 변경했다.
- 반품 요청 화면에서 요청 금액을 전체 금액 또는 직접 입력으로 선택할 수 있게 했다.
- 반품 요청 요약에는 선택한 증빙 이미지 수와 요청 금액이 즉시 반영되도록 했다.
- 푸터의 이용약관, 개인정보처리방침 링크는 고객센터 안내 대신 스크롤 가능한 본문 모달로 표시되도록 변경했다.
- DB와 사용자웹의 농장 표시는 `문경 햇살 농장` 하나로 고정했다.
- `farms` 테이블 중복 농장은 제거하고, 상품/수확 슬롯/조달/ML 예측 참조는 모두 동일한 farm_id를 바라보도록 정리했다.

주의:

- 반품 증빙 이미지는 현재 백엔드가 `evidence_image_url` 문자열 필드만 받는 구조라, 프론트에서 여러 장을 선택한 뒤 `/mock/returns/파일명` 경로들을 쉼표로 묶어 전달한다.
- 실제 파일 업로드 API가 추가되면 현재 파일 선택 로직은 유지하고, 서버에 전달하는 값만 업로드 결과 URL 목록으로 교체하면 된다.
- `dart analyze`는 로컬에서 오래 멈추는 문제가 있어 이번 마감 점검에서는 실행하지 않았고, 괄호/참조 중심의 정적 점검만 수행했다.
