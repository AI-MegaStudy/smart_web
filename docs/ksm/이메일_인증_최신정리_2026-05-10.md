# 이메일 인증 최신 정리

작성일: 2026-05-10

이 문서는 `docs/ksm/email` 폴더의 팀장 전달 내용을 `SMART_WEB` 작업 기준으로 요약한 최신 정리 문서다.

## 1. 결론

이메일 인증에서 문제가 됐던 DB 항목은 해결됐고, 실제 이메일 발송부터 회원가입까지 정상 확인된 상태다.

해결된 항목:

- `email_verifications.attempt_count` 추가
- `email_verifications.updated_at` 추가
- 가입 전 인증을 위해 `email_verifications.account_id`를 `NULL` 허용
- 해시 저장을 위해 `email_verifications.verification_code`를 `VARCHAR(255)`로 변경

따라서 백엔드 코드는 임시 제거했던 이메일 인증 필드와 실패 횟수 제한 로직을 다시 연결한 상태로 유지한다.

## 2. 백엔드 코드 기준

- `EmailVerification` 모델은 `TimestampMixin`을 사용해 `created_at`, `updated_at`을 매핑한다.
- `attempt_count`는 인증 코드 입력 실패 횟수 제한에 사용한다.
- 인증 코드는 평문 저장하지 않고 `verification_code` 컬럼에 해시로 저장한다.
- 가입 전 인증 시점에는 `account_id`가 비어 있을 수 있다.
- 회원가입 완료 후에는 인증 레코드의 `account_id`를 새 계정과 연결한다.

## 3. 환경변수 기준

백엔드 환경변수는 프로젝트 루트 `.env`를 기준으로 한다.

```text
C:\Users\User_KO\Documents\GitHub\smart_web\.env
```

기존 `backend/.env`가 남아 있더라도 실제 작업 기준은 루트 `.env`이다.

## 4. 확인할 API 흐름

```text
POST /api/v1/auth/email/send
→ POST /api/v1/auth/email/verify
→ POST /api/v1/auth/customers/signup
→ POST /api/v1/auth/login
```

농가 계정은 마지막 회원가입 API만 아래로 바꿔 같은 흐름을 사용한다.

```text
POST /api/v1/auth/owners/signup
```

## 5. QA 기준

정상 완료 확인:

- 이메일 인증 메일 발송 성공
- 메일로 받은 6자리 인증 코드 검증 성공
- 고객 회원가입 성공
- 회원가입 응답의 `email_verified`가 `true`
- 회원가입 응답의 `email_verification_required`가 `true`
- 로그인 성공

고객 화면에는 테스트 인증 코드, DB 필드명, API 경로, 내부 오류 문구를 직접 노출하지 않는다.

## 6. 새 계정 전체 흐름 QA

이메일 인증 완료 후 아래 사용자 흐름을 새 계정 기준으로 이어서 확인한다.

```text
회원가입
→ 로그인
→ 상품 상세
→ 예약함 담기
→ 예약 확인
→ 예약 생성
→ 주문서 작성
→ 카드 결제
→ 주문 완료
→ 주문 목록
→ 주문 상세
```

코드 연결 기준으로 필요한 API와 라우트는 모두 연결돼 있다.

고객 화면 문구 점검 결과:

- 고객 화면에 `Mock`, `더미`, `API`, `DB`, `테스트 코드`, `database integrity error`를 직접 표시하는 문자열은 발견되지 않았다.
- 코드 내부 클래스명과 파일명에는 `MockPaymentPage`, `MockAuthSession` 같은 개발용 이름이 남아 있으나 고객 화면 문구는 아니다.
