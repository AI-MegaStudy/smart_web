# 사용자 웹 필요한 정보 정리

이 문서는 `SMART_DOCS`의 기획서, 사용자 웹 프로토타입, API 설계서, 디자인 가이드를 바탕으로 `SMART_WEB`에서 구현해야 하는 사용자 웹 범위만 정리한 문서다.

## 0. 2026-05-11 최신 상태

고객웹 시연 마감 기준 최신 상태:

- 상품 품종 기준은 `신고`, `부사`이다.
- 농장명 표시는 `문경 햇살 농장` 하나로 고정한다.
- 고객 화면에는 `양광`, `신고 배`, `Harvest Demo Farm`, `QA 고객`, `customer@test.com`, `Mock 결제`, 승인금액 `0원` 같은 시연용 표현이 보이지 않도록 정리했다.
- 반품 요청 화면은 증빙 이미지 여러 장 선택과 요청 금액 직접 입력을 지원한다.
- 반품 요청 요약에는 증빙 이미지 선택 수와 요청 금액이 즉시 반영된다.
- 이용약관, 개인정보처리방침은 본문 모달로 표시한다.
- 고객 화면에는 ML 예측값을 직접 노출하지 않는다.

## 0.1 2026-05-10 상태

현재 사용자 웹은 상품, 예약, 주문, 결제, 주문 조회, 마이페이지, 회원정보 수정, 배송지 관리 API 연결을 완료했다.

보류 또는 추후 확인 항목:

- 이메일 인증은 DB에서 `attempt_count`, `updated_at`, `account_id NULL 허용`, `verification_code VARCHAR(255)` 보정이 완료된 상태다. 새 계정 기준 이메일 발송, 코드 검증, 회원가입, 로그인 QA를 다시 진행한다.
- 배송 조회와 반품 신청은 사용자 웹 코드 연결은 되어 있으나, 농가/관리자 앱에서 배송 상태 변경이 연결된 뒤 최종 성공 QA를 진행한다.
- 고객 화면에는 개발자용 `Mock`, `더미`, `API`, `DB`, 테스트 코드 문구를 직접 노출하지 않는다.

## 1. 사용자 웹 목표

사용자 웹은 고객이 농가 상품을 확인하고, 점주가 확정한 수확 슬롯을 기준으로 예약, 주문, 결제, 배송 확인, 반품 신청까지 진행하는 Flutter Web 서비스다.

고객 화면에는 다음 정보만 노출한다.

- 점주가 확정한 수확 예정 범위
- 점주가 확정한 예약 가능 수량
- 점주가 확정한 판매가
- 농장 배송 및 반품 정책

고객 화면에 ML 예측값은 직접 노출하지 않는다. ML 값은 점주가 수확 슬롯을 확정하기 위한 참고값이다.

## 2. 사용자 흐름

```text
홈
→ 상품 목록
→ 상품 상세
→ 수확 슬롯 선택
→ 로컬 예약함
→ 예약 확인
→ 서버 예약 생성
→ 주문서 작성
→ 카드 결제
→ 주문 완료
→ 주문 상세 확인
→ 배송 확인
→ 반품 신청
```

역할 기준은 다음과 같다.

- Model: API 응답, 요청 DTO, 화면에 필요한 데이터 구조를 정의한다.
- View: Flutter 화면과 위젯만 담당한다. API를 직접 호출하지 않는다.
- ViewModel: 화면 상태, 입력값 검증, 버튼 클릭 흐름, Repository 호출을 담당한다.
- Repository: API 호출을 감싸고 ViewModel이 백엔드 구조에 과하게 의존하지 않게 한다.

## 6. 백엔드 담당자에게 전달해야 하는 연결 지점

### 인증

- 회원가입: `POST /auth/customers/signup`
- 이메일 인증 발송: `POST /auth/email/send`
- 이메일 인증: `POST /auth/email/verify`
- 인증 코드 재발송: `POST /auth/email/resend`
- 이메일 인증 상태: `GET /auth/email/status`
- 로그인: `POST /auth/login`
- 내 정보: `GET /me`
- 로그인 성공 시 `access_token`을 반환해야 한다.
- 인증 필요 API는 `Authorization: Bearer <access_token>` 헤더를 사용한다.

### 상품과 슬롯

- 홈 추천 상품: `GET /products?featured=true`
- 상품 목록: `GET /products`
- 상품 상세: `GET /products/{product_id}`
- 상품 슬롯: `GET /products/{product_id}/slots`
- 상품 목록은 `product_status=ACTIVE`, `slot_status=OPEN`인 슬롯이 있는 상품을 우선 보여준다.
- 고객 웹에서는 `product_id`, `slot_id`, `package_unit_kg`, `package_count`, `confirmed_price`, `confirmed_harvest_start`, `confirmed_harvest_end`, `available_kg`가 필요하다.

### 로컬 예약함

- 로컬 예약함은 서버에 바로 저장하지 않는다.
- 저장 데이터는 브라우저 저장소와 Flutter 상태로 관리한다.
- 서버 기록은 `POST /reservations` 성공 시점부터 시작한다.
- 수량 검증과 가격 검증은 서버 예약 생성 시 최종 확인해야 한다.

로컬 예약함 데이터 예시:

```json
{
  "items": [
    {
      "slotId": 12,
      "productId": 3,
      "productName": "후지 사과 5kg",
      "farmName": "청송 햇살농원",
      "packageUnitKg": 5.0,
      "packageCount": 2,
      "unitPrice": 39000,
      "reservedKg": 10.0
    }
  ]
}
```

### 예약

- 예약 미리보기: `POST /reservations/preview`
- 예약 생성: `POST /reservations`
- 요청 기준은 `slot_id`와 `package_count`다.
- `reserved_kg`는 프론트에서 직접 입력받지 않고 `package_count * package_unit_kg`로 계산한다.
- 서버는 슬롯 행 잠금, 남은 수량 검증, 가격 검증, 예약 만료 시간 생성이 필요하다.

예약 생성 요청 예시:

```json
{
  "items": [
    {
      "slot_id": 12,
      "package_count": 2
    }
  ]
}
```

### 주문

- 주문 생성: `POST /orders/from-reservation`
- 요청은 `reservation_id`, 수령인, 전화번호, 배송 주소, 배송 메모를 포함한다.
- 예약 상태가 `RESERVED`이고 만료 전이어야 주문 생성이 가능하다.

### 결제

- 카드 결제 승인: `POST /payments/mock-approve`
- 요청은 `order_id`, `idempotency_key`를 포함한다.
- 결제 승인 후 주문 상태는 `PROCUREMENT_REQUESTED`로 이동해야 한다.
- 결제 승인 후 점주 앱 쪽 발주 데이터가 생성되어야 한다.

### 주문 조회와 배송

- 내 주문 목록: `GET /me/orders`
- 내 주문 상세: `GET /me/orders/{order_id}`
- 배송 정보: `GET /me/orders/{order_id}/shipment`
- 주문 상세 화면에는 주문, 결제, 발주, 선별, 배송 상태 타임라인이 필요하다.

### 반품

- 반품 신청: `POST /returns`
- 내 반품 목록: `GET /me/returns`
- 반품 신청은 배송 완료 주문에서 가능하게 처리한다.
- 프론트는 사유 코드, 상세 사유, 증빙 이미지, 요청 금액을 전달한다.

## 8. 디자인 기준

디자인 키워드는 신선함, 정직한 농가, 예약 안정성, 수확 직배송, 명확한 상태 안내다.

컬러는 다음 용도로 사용한다.

- Primary Green: 주요 버튼, 농장, 신선함
- Warm Yellow: 수확, 예약 강조
- Neutral Gray: 배경, 구분선, 본문
- Alert Red: 반품, 거절, 주의 상태
- Info Blue: 상태 안내

공통 컴포넌트는 다음을 우선 만든다.

- `StatusBadge`
- `NoticeBox`
- `ProductCard`
- `HarvestSlotCard`
- `PriceText`
- `FormSection`

## 9. 고객 화면 문구 원칙

아래 문구를 상품 상세, 예약 확인, 주문 상세에 상황에 맞게 사용한다.

```text
농가가 확정한 수확 예정 범위입니다.
기상과 생육 상황에 따라 일정이 조정될 수 있습니다.
```

```text

수확 일정은 기상과 생육 상황에 따라 조정될 수 있으며, 변동 시 주문 상세에서 안내됩니다.
```

## 10. 추가하면 좋은 기능 제안

기획서 범위를 크게 벗어나지 않는 선에서 사용자 웹에 추가하면 좋은 기능은 다음이다.

- 예약 만료 카운트다운: `reservations.reserved_until` 기준으로 주문서 작성 화면에 남은 시간을 표시한다.
- 배송지 기본값 선택: `customer_profiles.default_*` 값을 먼저 보여주고 수정 가능하게 한다.
- 예약 가능 수량 경고: 남은 수량이 적은 슬롯에 “잔여 수량 적음” 배지를 표시한다.
- 주문 상태 타임라인: 주문 상세에서 결제, 농가 확인, 선별, 배송, 배송 완료를 한눈에 보여준다.
- 반품 가능 조건 안내: 배송 완료 상태에서만 반품 버튼을 활성화하고, 그 외 상태에서는 안내 문구를 보여준다.

이 기능들은 백엔드와 새 테이블을 크게 추가하지 않아도 구현 가능하며, 기존 API 응답에 필드 몇 개만 명확히 포함되면 된다.

## 11. Codex와 huashu-design 사용 방법

현재 `SMART_DOCS/huashu-design-codex-usage.md`에 따르면 `huashu-design`은 실제 운영용 Flutter 코드 작성보다는 HTML 기반 디자인 시안, 발표용 프로토타입, 디자인 리뷰에 적합하다.

사용 예시:

```text
huashu-design으로 사용자 웹 상품 상세 화면을 기획서 기준으로 디자인 리뷰해줘. 아직 Flutter 코드는 수정하지 말고 개선 우선순위만 정리해줘.
```

```text
huashu-design으로 사용자 웹 예약 확인 화면의 HTML 프로토타입을 더 발표용으로 다듬어줘. Playwright로 데스크톱과 모바일 화면을 검증해줘.
```

실제 `SMART_WEB` Flutter 구현을 할 때는 `huashu-design`보다 `AGENTS.md`와 이 문서를 기준으로 작업하는 편이 맞다.

추가 화면의 백엔드 연결 지점은 아래 문서를 기준으로 한다.

```text
docs/ksm/마이페이지_배송지_회원정보_API연결정리.md
```

## 2026-05-06 결제/주문 흐름 최신화

결제 화면은 단순 결제 수단 선택 화면이 아니라 수확 예약 서비스에 맞는 `결제 전 확인` 화면으로 정리했다.

결제 화면에서 표시하는 정보:

- 수확 예약 상품
- 수확 예정 범위
- 박스 수량
- 총 중량
- 배송 정보
- 결제 수단
- 상품 금액
- 배송비
- 최종 결제 금액
- 수확 일정 변동 가능성 안내
- 결제 전 확인 동의

결제 완료 후 주문 상태는 바로 배송 완료가 아니라 `농가 확인 중`으로 표시한다.

주문 상세에서는 아래 흐름으로 상태를 안내한다.

```text
결제 완료
→ 농가 확인 중
→ 선별 완료
→ 배송 중
→ 배송 완료
→ 반품 요청 접수
```

반품 신청은 `배송 완료` 상태일 때만 표시한다.
