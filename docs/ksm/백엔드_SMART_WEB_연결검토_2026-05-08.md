# SMART_WEB - HARVEST-SLOT-BACKEND 연결 검토

작성일: 2026-05-08  
최신 확인: 2026-05-10, `SMART_WEB/backend` 내부 백엔드 기준 재확인
목적: 사용자 웹 `SMART_WEB`과 백엔드 `HARVEST-SLOT-BACKEND` 연결 가능 여부를 백엔드 담당자에게 공유하기 위한 정리

## 1. 결론

현재 백엔드는 `SMART_WEB` 사용자 예약 웹과 연결 가능합니다.

2026-05-09 기준으로 `C:\Users\User_KO\Documents\GitHub\smart_web\backend` 안에 백엔드 코드가 포함되어 있으며, 기존에 확인했던 `HARVEST-SLOT-BACKEND`와 같은 FastAPI + MySQL 구조입니다.

핵심 사용자 흐름인 상품 조회, 수확 슬롯 조회, 예약 생성, 주문 생성, 카드 결제 승인 API, 주문 조회, 배송 조회, 반품 신청 API가 대부분 구현되어 있습니다.

다만 바로 연결하기 전에 프론트와 백엔드가 맞춰야 할 부분이 있습니다.

- 모든 응답이 `data`, `message`, `error` 구조로 감싸져 있으므로 프론트 Repository에서 `data`를 꺼내 사용해야 합니다.
- 배송지 관련 필드는 일부만 있었으나, 2026-05-10 기준 프론트 담당자가 MVP용 배송지 API를 추가했습니다.
- 회원정보 수정 API도 2026-05-10 기준 `PUT /api/v1/me`로 추가했습니다.
- 이메일 인증은 `email_verifications.attempt_count`, `updated_at`, `account_id NULL 허용`, `verification_code VARCHAR(255)` DB 보정 및 테스트가 완료되어 재연결 기준으로 정리합니다.
- 고객센터/푸터 정보는 DB/API 없이 프론트 더미 데이터로 유지하기로 결정했습니다.
- 일부 문서, 스키마 예시, 주석에서 한글 깨짐이 보여 Swagger 문서 품질 정리가 필요합니다.

## 3. 공통 응답 구조

백엔드는 모든 일반 API 응답을 아래 구조로 반환합니다.

```json
{
  "data": {},
  "message": "success",
  "error": null
}
```

따라서 프론트에서는 실제 화면 데이터로 `response.data.data` 또는 Repository 내부 변환 결과를 사용해야 합니다.

에러 응답도 같은 구조를 사용합니다.

```json
{
  "data": {},
  "message": "validation failed",
  "error": []
}
```

프론트에서는 `message`와 `error`를 사용자용 한글 메시지로 변환해서 표시하는 방식이 필요합니다.

## 3.1 로컬 백엔드 실행 기준

`SMART_WEB` 루트에서 백엔드를 실행할 때는 아래 명령을 기준으로 봅니다.

```bash
uvicorn backend.app.main:app --reload
```

Swagger 확인 주소:

```text
http://localhost:8000/docs
```

프론트 API Base URL 개발 기본값:

```text
http://localhost:8000/api/v1
```

DB는 `backend/app/core/config.py` 기준 MySQL 연결입니다. 환경변수 파일은 프로젝트 루트 `.env`를 기준으로 읽습니다.

```text
mysql+pymysql://<user>:<password>@<host>:3306/harvest_slot_db?charset=utf8mb4
```

로컬 MySQL, 팀 공용 MySQL, AWS RDS MySQL 모두 환경변수 설정으로 연결 가능합니다.

## 8. 백엔드 추가 또는 정리가 필요한 부분

### 8.1 배송지 관리 API

현재 `CustomerProfile`에 기본 배송지 성격의 필드는 있습니다.

- `default_receiver_name`
- `default_receiver_phone`
- `default_shipping_address`

하지만 `SMART_WEB`에 구현된 배송지 관리 화면은 여러 배송지 추가, 수정, 기본 배송지 설정 흐름을 전제로 합니다.

팀장 컨펌 결과, 배송지 관련 필드는 일부 있으나 독립 API가 없는 상태이므로 프론트 담당자가 백엔드에 직접 API를 추가하기로 했습니다.

우선 추가할 API 후보는 아래와 같습니다.

```http
GET /api/v1/me/addresses
POST /api/v1/me/addresses
PUT /api/v1/me/addresses/{address_id}
DELETE /api/v1/me/addresses/{address_id}
PATCH /api/v1/me/addresses/{address_id}/default
```

권장 필드:

```json
{
  "address_id": 1,
  "label": "내집",
  "receiver_name": "홍길동",
  "receiver_phone": "010-1111-2222",
  "zonecode": "06236",
  "address": "서울시 강남구 테헤란로 123",
  "detail_address": "101동 1001호",
  "delivery_memo": "문 앞에 놓아주세요",
  "is_default": true,
  "last_used_at": "2026-05-08T12:00:00"
}
```

### 8.2 사용자 정보 수정 API

마이페이지 회원정보 화면 연결을 위해 아래 API가 있으면 좋습니다.

```http
PUT /api/v1/me
```

수정 가능 범위:

- 이름
- 전화번호
- 이메일 변경 요청
- 마케팅 수신 동의

이메일 변경은 인증이 필수가 아니어도 저장 가능하게 하되, 인증 여부는 `email_verified`로 표시하면 됩니다.

### 8.3 고객센터/푸터 정보

현재 프론트에는 고객센터 전화 연결과 이메일 문의가 정적 데이터로 들어가 있습니다.

팀장 컨펌 결과, 푸터/고객센터 정보는 DB에 없는 상태이며 이번 범위에서는 별도 API를 만들지 않고 프론트 더미 데이터로 유지하기로 결정했습니다.

따라서 `GET /api/v1/service-info` 같은 설정 API는 이번 작업 범위에서 제외합니다.

프론트에서 유지할 정보:

- 고객센터 전화번호
- 이메일 문의 주소
- 운영 시간
- 푸터 안내 문구
- 전화 연결, 이메일 연결 동작

## 9.1 2026-05-09 상품 API 1차 연결 결과

프론트에서 상품/수확 슬롯 API를 실제 백엔드에 1차 연결했다.

추가 또는 변경한 프론트 구조:

- `lib/core/api/api_client.dart`
- `lib/core/api/api_exception.dart`
- `lib/data/repositories/product_api_repository.dart`
- `ProductRepositoryContract` 기반 ViewModel 주입 구조 유지

연결한 API:

```http
GET /api/v1/products?featured=true
GET /api/v1/products
GET /api/v1/products/{product_id}
GET /api/v1/products/{product_id}/slots
```

확인 결과:

- 로컬 백엔드 `http://127.0.0.1:8000/api/v1` 기준으로 상품 목록과 상품 상세 데이터를 받을 수 있다.
- 상품 카드의 수확 예정일, 잔여 수량, 가격은 상품 응답이 아니라 슬롯 조회 결과를 우선 사용한다.
- 상품명은 백엔드 `product_name` 원문을 고객 화면에 표시한다.
- 상품 상세의 예약 가능 여부는 `GET /products/{product_id}/slots` 결과를 기준으로 판단한다.
- 상품 상세 응답의 `open_slot_count`가 0이더라도 실제 슬롯 조회 결과가 있으면 예약 가능 화면으로 표시한다.
- 슬롯이 없거나 잔여 수량이 0인 상품은 `수확 일정 준비중`, `확정 후 예약 가능` 안내로 표시한다.

주의할 점:

- 현재 백엔드 더미 데이터에는 `product_name`이 `예약 사과 5kg`처럼 들어올 수 있다. 고객에게 보여줄 최종 상품명 정책은 백엔드 데이터 입력 기준으로 맞추는 것이 좋다.
- 백엔드 품종 값은 현재 `Fuji`, `Shingo`처럼 영어 또는 임시값이 들어올 수 있다. 프론트에서는 `Fuji -> 부사`, `Shingo -> 신고`, `yanggwang/yangkwang -> 신고`로 매핑한다.
- 2026-05-11 고객웹 시연 마감 기준 사과 품종은 `신고`, `부사` 중심으로 맞춘다. 과거 `양광`, `부사` 기준은 당시 작업 기록으로만 본다.
- `flutter analyze` 기준 문법/정적 분석은 통과했다.
- `dart format`은 Windows/IDE Dart 분석 서버 프로세스가 남아 오래 걸리는 현상이 있어, 필요 시 Dart/Flutter 프로세스 정리 후 단독 실행한다.

다음 연결 순서는 인증, 예약 생성, 주문/결제, 주문 조회, 반품 순서가 적절합니다.

다만 마이페이지의 배송지 관리는 독립 API가 없으므로 프론트 담당자가 백엔드에 직접 API를 추가한 뒤 연결합니다.

푸터/고객센터 정보는 DB/API 없이 프론트 더미 데이터로 유지합니다.
