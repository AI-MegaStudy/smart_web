# 사용자 웹 추가 기능 제안

이 문서는 `SMART_WEB` 사용자 웹 구현 후 완성도를 높이기 위해 추가할 수 있는 기능을 정리한 문서다.

아래 기능들은 기존 기획서 범위에서 벗어나지 않고, 백엔드에 큰 구조 변경을 요구하지 않는 항목만 선별했다.

## 1. 추가 기능 선정 기준

추가 기능은 아래 조건을 만족할 때만 진행한다.

- 사용자 예약, 주문, 배송 확인 흐름을 더 명확하게 만든다.
- 기존 API 응답 필드 또는 작은 필드 추가만으로 구현할 수 있다.
- 백엔드 담당자에게 필요한 연결 지점을 설명할 수 있다.
- 관리자 앱, 머신러닝, 딥러닝 담당자의 범위를 침범하지 않는다.
- 고객 화면에 개발자용 상태 코드, DB 테이블명, ML 예측값을 직접 노출하지 않는다.

## 2. 우선 추천 기능

| 우선순위 | 기능 | 적용 화면 | 백엔드 부담 | 추천 이유 |
|---|---|---|---|---|
| 1 | 예약 만료 카운트다운 | 예약 확인, 주문서 작성 | 낮음 | 예약 점유 시간이 있는 서비스 특성을 고객이 바로 이해할 수 있다. |
| 2 | 기본 배송지 선택 | 주문서 작성 | 낮음 | 주문 입력 시간을 줄이고 실제 서비스처럼 보인다. |
| 3 | 잔여 수량 적음 배지 | 상품 목록, 상품 상세 | 낮음 | 예약 가능 수량을 강조해 구매 결정을 돕는다. |
| 4 | 주문 상태 타임라인 | 주문 상세 | 낮음 | 결제, 농가 확인, 선별, 배송 흐름을 한눈에 보여준다. |
| 5 | 반품 가능 조건 안내 | 주문 상세, 반품 신청 | 낮음 | 반품 가능 시점과 조건을 명확히 안내한다. |

## 3. 예약 만료 카운트다운

### 목적

서버 예약 생성 후 고객이 주문서 작성과 결제를 일정 시간 안에 완료해야 한다는 점을 명확히 보여준다.

### 적용 화면

- `/reservation/confirm`
- `/checkout/:reservationId`

### 화면 동작

- 예약 생성 성공 후 서버가 내려준 `reserved_until` 시간을 기준으로 남은 시간을 표시한다.
- 남은 시간이 5분 이하이면 주의 색상으로 강조한다.
- 만료되면 주문 생성 버튼을 비활성화하고 예약함 또는 상품 상세로 돌아가도록 안내한다.

### 백엔드에 요청할 내용

`POST /reservations` 응답에 아래 값이 필요하다.

```json
{
  "reservation_id": 5,
  "reservation_status": "RESERVED",
  "reserved_until": "2026-10-12T14:30:00"
}
```

`POST /orders/from-reservation`에서는 예약 만료 여부를 다시 검증해야 한다.

### 고객 화면 문구 예시

```text
예약 수량을 잠시 확보했습니다.
남은 시간 안에 주문서를 작성해 주세요.
```

```text
예약 시간이 만료되었습니다.
상품 상세에서 다시 예약해 주세요.
```

## 4. 기본 배송지 선택

### 목적

주문서 작성 시 고객 프로필에 저장된 기본 배송 정보를 먼저 보여주고, 필요한 경우 수정하게 한다.

### 적용 화면

- `/checkout/:reservationId`

### 화면 동작

- 주문서 작성 화면 진입 시 기본 배송지 정보를 불러온다.
- 기본값을 그대로 사용하거나 직접 수정할 수 있다.
- 배송 메모는 선택형 목록과 직접 입력을 함께 제공한다.

### 백엔드에 요청할 내용

`GET /me` 또는 별도 고객 프로필 응답에 아래 값이 필요하다.

```json
{
  "customer_profile": {
    "name": "홍길동",
    "phone": "010-1111-2222",
    "default_receiver_name": "홍길동",
    "default_receiver_phone": "010-1111-2222",
    "default_shipping_address": "서울시 강남구 ..."
  }
}
```

`POST /orders/from-reservation` 요청에는 기존 기획서처럼 실제 주문 배송 정보를 보낸다.

```json
{
  "reservation_id": 5,
  "receiver_name": "홍길동",
  "receiver_phone": "010-1111-2222",
  "shipping_address": "서울시 강남구 ...",
  "delivery_memo": "문 앞에 놓아주세요"
}
```

### 고객 화면 문구 예시

```text
기본 배송지를 사용합니다.
```

```text
이번 주문에서만 배송지를 수정합니다.
```

## 5. 잔여 수량 적음 배지

### 목적

예약 가능한 수량이 적은 상품이나 슬롯을 고객이 빠르게 알아볼 수 있게 한다.

### 적용 화면

- `/products`
- `/products/:id`

### 화면 동작

- 남은 예약 가능 수량이 특정 기준 이하이면 배지를 표시한다.
- 예시 기준은 `available_kg <= package_unit_kg * 5` 또는 `available_package_count <= 5`다.
- 수량이 0이면 예약 불가 상태로 표시하고 담기 버튼을 비활성화한다.

### 백엔드에 요청할 내용

상품 목록과 슬롯 응답에 아래 값이 필요하다.

```json
{
  "slot_id": 12,
  "confirmed_reservable_kg": 300.0,
  "reserved_kg": 275.0,
  "available_kg": 25.0,
  "package_unit_kg": 5.0,
  "available_package_count": 5,
  "slot_status": "OPEN"
}
```

### 고객 화면 문구 예시

```text
잔여 수량 적음
```

```text
예약 마감
```

## 6. 주문 상태 타임라인

### 목적

주문 상세에서 결제, 농가 확인, 선별, 배송, 배송 완료 상태를 한눈에 보여준다.

### 적용 화면

- `/me/orders/:orderId`

### 화면 동작

- 현재 주문 상태까지는 완료 상태로 표시한다.
- 다음 단계는 예정 상태로 표시한다.
- 배송 중이면 택배사와 송장번호를 함께 보여준다.
- 반품 요청 또는 환불 상태가 있으면 타임라인 하단에 별도 상태를 추가한다.

### 백엔드에 요청할 내용

`GET /me/orders/{order_id}` 응답에 주문 상태, 결제 상태, 발주 상태, 배송 상태, 반품 상태가 포함되어야 한다.

```json
{
  "order_id": 8,
  "order_number": "ORD-20261012-008",
  "order_status": "SHIPPED",
  "payment": {
    "payment_status": "APPROVED",
    "approved_at": "2026-10-12T14:02:00"
  },
  "procurement": {
    "procurement_status": "APPROVED",
    "decided_at": "2026-10-12T16:20:00"
  },
  "shipment": {
    "shipment_status": "SHIPPED",
    "carrier_name": "CJ대한통운",
    "tracking_number": "5891-1202-4810",
    "shipped_at": "2026-10-18T09:40:00"
  },
  "return_request": null
}
```

### 고객 화면 상태 표시

| 서버 상태 | 고객 표시 |
|---|---|
| `PAYMENT_PENDING` | 결제 대기 |
| `PAID` | 결제 완료 |
| `PROCUREMENT_REQUESTED` | 농가 확인 중 |
| `PROCUREMENT_APPROVED` | 농가 승인 완료 |
| `QUALITY_CHECKING` | 선별 중 |
| `READY_TO_SHIP` | 발송 준비 |
| `SHIPPED` | 배송 중 |
| `DELIVERED` | 배송 완료 |
| `RETURN_REQUESTED` | 반품 요청 접수 |
| `REFUNDED` | 환불 완료 |

### 고객 화면 문구 예시

```text
농가가 수확 가능 수량을 확인하고 있습니다.
```

```text
선별이 완료되어 발송을 준비하고 있습니다.
```

## 7. 반품 가능 조건 안내

### 목적

반품 신청이 가능한 조건을 고객에게 명확히 안내하고, 불가능한 상태에서는 버튼을 비활성화한다.

### 적용 화면

- `/me/orders/:orderId`
- `/me/orders/:orderId/return`

### 화면 동작

- 주문 상태가 `DELIVERED`일 때 반품 신청 버튼을 활성화한다.
- 이미 `RETURN_REQUESTED` 상태이면 접수 완료 상태를 보여준다.
- `SHIPPED` 이전 상태에서는 반품 신청 대신 주문 진행 상태 안내를 보여준다.
- 반품 신청 화면에서는 사유 코드, 상세 사유, 증빙 이미지, 요청 금액을 입력 또는 선택한다.

### 백엔드에 요청할 내용

`GET /me/orders/{order_id}` 응답에 반품 가능 여부를 내려주면 프론트 구현이 단순해진다.

```json
{
  "order_id": 8,
  "order_status": "DELIVERED",
  "returnable": true,
  "return_deadline": "2026-10-25T23:59:59"
}
```

`POST /returns` 요청은 아래 구조를 기준으로 한다.

```json
{
  "order_id": 8,
  "reason_code": "DAMAGED",
  "reason_detail": "배송 중 일부 상품이 눌렸습니다.",
  "request_amount": 39000,
  "evidence_image_urls": [
    "https://example.com/evidence-1.jpg"
  ]
}
```

### 고객 화면 문구 예시

```text
배송 완료 후 반품 신청이 가능합니다.
```

```text
반품 요청이 접수되었습니다.
농가 확인 후 환불 진행 상태를 안내해 드립니다.
```

## 9. 백엔드 담당자에게 전달할 요약

추가 기능을 위해 백엔드에 요청할 핵심 필드는 다음이다.

- 예약: `reservation_id`, `reservation_status`, `reserved_until`
- 슬롯: `slot_id`, `available_kg`, `available_package_count`, `package_unit_kg`, `slot_status`
- 고객 프로필: `default_receiver_name`, `default_receiver_phone`, `default_shipping_address`
- 주문 상세: `order_status`, `payment`, `procurement`, `shipment`, `return_request`
- 반품 가능 여부: `returnable`, `return_deadline`

위 필드가 있으면 프론트는 별도 복잡한 계산 없이 사용자에게 명확한 예약, 주문, 배송, 반품 상태를 보여줄 수 있다.
