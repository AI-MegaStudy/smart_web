# SMART_WEB 내부 백엔드 구조 및 연동 정리

작성일: 2026-05-09  
대상 경로: `C:\Users\User_KO\Documents\GitHub\smart_web\backend`

## 1. 결론

`SMART_WEB` 프로젝트 안에 새로 들어온 `backend` 폴더는 FastAPI 기반 백엔드입니다.

기존에 따로 확인했던 `HARVEST-SLOT-BACKEND` 구조와 거의 동일하며, 사용자 웹에서 필요한 핵심 API 흐름과 연결 가능합니다.

현재 기준으로 프론트 연결은 다음 방식으로 진행하면 됩니다.

```text
SMART_WEB Flutter Web
→ FastAPI 서버
→ MySQL DB
```

2026-05-10 최신 상태:

- 상품, 슬롯, 로그인, `/me`, 예약, 주문, 결제, 주문 목록, 주문 상세 API는 사용자 웹에서 연결했다.
- 회원정보 수정용 `PUT /api/v1/me`를 백엔드에 추가하고 프론트에서 연결했다.
- 배송지 관리용 `/api/v1/me/addresses` 계열 API를 백엔드에 추가하고 프론트에서 연결했다.
- 이메일 인증은 DB에서 `attempt_count`, `updated_at`, `account_id NULL 허용`, `verification_code VARCHAR(255)` 보정이 완료되어 재연결 기준으로 정리했다.
- 배송/반품 최종 성공 검증은 농가/관리자 앱에서 배송 상태를 변경하는 흐름이 연결된 뒤 진행한다.

## 2. 백엔드 기술 구조

| 구분 | 내용 |
|---|---|
| 서버 프레임워크 | FastAPI |
| 라우터 위치 | `backend/app/routers/` |
| 서비스 위치 | `backend/app/services/` |
| 모델 위치 | `backend/app/models/` |
| 스키마 위치 | `backend/app/schemas/` |
| DB 접근 | SQLAlchemy |
| DB 종류 | MySQL 계열 |
| DB 드라이버 | `pymysql` |
| API Base URL | `/api/v1` |
| 인증 방식 | JWT Bearer Token |

DB 연결 문자열은 `backend/app/core/config.py`에서 아래 형태로 조합됩니다.

```text
mysql+pymysql://<user>:<password>@<host>:3306/harvest_slot_db?charset=utf8mb4
```

따라서 로컬 MySQL, 팀 공용 MySQL, AWS RDS MySQL 모두 환경변수만 맞으면 연결 가능합니다.

## 3. FastAPI 실행 기준

`smart_web` 루트에서 실행할 때 기준 명령은 아래와 같습니다.

```bash
uvicorn backend.app.main:app --reload
```

실행 후 확인 주소:

```text
http://localhost:8000/docs
```

프론트에서 API를 호출할 때 기본 주소 예시는 아래처럼 잡으면 됩니다.

```text
http://localhost:8000/api/v1
```

운영 또는 배포 환경에서는 이 주소만 배포된 API 서버 주소로 바꾸면 됩니다.

## 4. 환경변수 기준

백엔드는 프로젝트 루트 `.env`를 읽도록 설정되어 있습니다.

주요 설정값:

```env
DATABASE_HOST=your-db-host
DATABASE_PORT=3306
DATABASE_USER=your-db-user
DATABASE_PASSWORD=your-db-password
DATABASE_NAME=harvest_slot_db
DATABASE_CHARSET=utf8mb4

JWT_SECRET_KEY=change-this-secret
ACCESS_TOKEN_EXPIRE_MINUTES=60
CORS_ALLOWED_ORIGINS=*

EMAIL_DEV_MODE=true
EMAIL_VERIFICATION_REQUIRED=true
```

프론트 개발 중에는 `CORS_ALLOWED_ORIGINS=*` 또는 Flutter Web 개발 서버 주소를 허용해야 합니다.

참고:

- `EMAIL_VERIFICATION_REQUIRED=true` 기준으로 가입 전 이메일 인증 후 회원가입 흐름을 사용한다.
- 기존 `backend/.env`가 남아 있어도 현재 작업 기준은 `smart_web/.env`이다.

## 6. 프론트 Repository 연결 규칙

백엔드 응답은 공통적으로 아래 구조입니다.

```json
{
  "data": {},
  "message": "success",
  "error": null
}
```

따라서 `SMART_WEB`에서는 Repository 계층에서 `data`만 꺼내 화면 모델로 변환해야 합니다.

View 또는 ViewModel에서 직접 `response["data"]`를 반복해서 처리하지 말고, Repository에서 백엔드 응답 구조를 숨기는 방향이 좋습니다.

권장 구조:

```text
View
→ ViewModel
→ Repository Contract
→ Api Repository
→ FastAPI
```

## 7. 현재 부족하거나 추가 논의가 필요한 부분

### 7.1 배송지 관리 API

2026-05-10 기준 `/api/v1/me/addresses` 계열 API를 MVP 방식으로 추가했습니다.

현재 `CustomerProfile`에는 기본 배송지 필드가 있습니다.

- `default_receiver_name`
- `default_receiver_phone`
- `default_shipping_address`

하지만 사용자 웹의 배송지 관리 화면은 여러 배송지 추가, 수정, 기본 배송지 설정을 전제로 합니다.

필요 API 후보:

```http
GET /api/v1/me/addresses
POST /api/v1/me/addresses
PUT /api/v1/me/addresses/{address_id}
DELETE /api/v1/me/addresses/{address_id}
PATCH /api/v1/me/addresses/{address_id}/default
```

현재 구현은 별도 배송지 테이블 없이 `customer_profiles`의 기본 배송지 필드 하나를 주소 목록처럼 반환합니다.

추후 여러 배송지를 정식 지원하려면 별도 배송지 테이블과 API 재설계가 필요합니다.

### 7.2 회원정보 수정 API

2026-05-10 기준 사용자 웹 연결을 위해 `PUT /api/v1/me`를 추가했습니다.

필요 API 후보:

```http
PUT /api/v1/me
```

수정 가능 필드 후보:

- 이름
- 전화번호
- 이메일은 현재 수정 대상에서 제외하고 조회값으로 표시합니다.
- 마케팅 수신 동의는 현재 범위에서 제외했습니다.

이메일 변경 시 인증 여부는 `email_verified`로 별도 표시하는 방식이 좋습니다.

### 7.3 문서와 일부 코드의 한글 깨짐

아래 파일들에서 한글 깨짐이 보입니다.

- `backend/app/main.py`

백엔드 담당자에게 UTF-8 기준으로 정리 요청하는 것이 좋습니다.

## 9. 백엔드 담당자에게 전달할 요약

프론트 사용자 웹 기준으로 핵심 API는 대부분 연결 가능합니다.

다만 아래 항목은 백엔드 쪽 확인 또는 추가 논의가 필요합니다.

1. 새 계정 기준 이메일 인증 → 회원가입 → 로그인 전체 QA
2. 여러 배송지를 정식 관리할 별도 배송지 테이블 필요 여부 결정
3. Swagger 문서와 스키마 예시의 한글 깨짐 정리 필요

프론트는 상품, 슬롯, 로그인, 이메일 인증, 예약, 주문, 결제, 주문 조회, 마이페이지, 배송지 흐름을 실제 API Repository로 연결했습니다. 배송/반품 최종 성공 검증은 농가/관리자 앱 배송 상태 변경 흐름 연결 후 진행합니다.
