## 4. 더미 데이터 최종 통일

화면별 더미 데이터가 아직 완전히 같은 기준으로 관리되는 것은 아니다.

정리할 항목:

- 상품명
- 농장명
- 수확 예정 날짜
- 패키지 단위
- 잔여 수량
- 주문번호
- 주문 상태
- 배송 상태
- 결제 금액
- 반품 가능 주문 상태
- 회원 이름, 이메일, 휴대폰번호
- 기본 배송지

권장 기준:

```text

대표 상품: 양광 사과
대표 농장: 청송 햇살농원
대표 수확 예정: 10.12-10.18
대표 회원: 홍길동
대표 이메일: customer@test.com
대표 연락처: 010-1111-2222
대표 배송지: 서울시 강남구 테헤란로 123
대표 택배사: CJ대한통운
대표 송장번호: 5891-1202-4810
결제 직후 상태: 농가 확인 중
```

## 5. 빈 상태/오류 상태 보강

현재는 대부분 더미 데이터가 있는 상태를 기준으로 화면이 구성되어 있다.

추가하면 좋은 상태:

- 예약함이 비어 있을 때
- 주문 목록이 비어 있을 때
- 배송지 목록이 비어 있을 때
- 상품 목록이 비어 있을 때
- 상품 이미지 로딩 실패
- API 로딩 실패
- 네트워크 오류

고객 화면에는 API 경로, DB명, 내부 상태 코드가 노출되지 않도록 한다.

## 7. 실제 API 연동 전 준비

실제 API 연동 전 아래 구조를 정리하면 좋다.

- 더미 Repository와 실제 API Repository 분리
- 공통 API 클라이언트 정리
- 인증 토큰 저장 위치 정리
- API 오류 메시지 고객용 문구로 변환
- 로딩/성공/실패 상태 ViewModel 기준 통일
- 라우트 arguments 타입 정리

### Repository 교체 기준

2026-05-08 기준으로 ViewModel은 아래 Repository 인터페이스를 바라보도록 정리했다.

```text
ProductRepositoryContract
LocalBasketRepositoryContract
OrderRepositoryContract
```

현재 구현체:

| 현재 구현체 | 역할 | 실제 API 연결 시 교체 방향 |
|---|---|---|
| `ProductRepository` | 홈 대표 상품, 상품 목록, 상품 상세, 수확 슬롯 더미 데이터 제공 | `ApiProductRepository`로 교체 |
| `LocalBasketRepository` | 예약 생성 전 로컬 예약함 상태 관리 | 브라우저 저장소 기반 Repository로 유지하거나 `LocalStorageBasketRepository`로 분리 |
| `OrderRepository` | 주문 목록, 주문 상세, 반품 요청 더미 상태 제공 | `ApiOrderRepository`와 `ApiReturnRepository`로 분리 가능 |

ViewModel 연결 기준:

| ViewModel | 사용하는 인터페이스 |
|---|---|
| `HomeViewModel` | `ProductRepositoryContract` |
| `ProductListViewModel` | `ProductRepositoryContract` |
| `ProductDetailViewModel` | `ProductRepositoryContract`, `LocalBasketRepositoryContract` |
| `LocalBasketViewModel` | `LocalBasketRepositoryContract` |
| `ReservationConfirmViewModel` | `LocalBasketRepositoryContract`, `ProductRepositoryContract` |
| `CheckoutViewModel` | `LocalBasketRepositoryContract` |
| `PaymentViewModel` | `LocalBasketRepositoryContract` |
| `OrderCompleteViewModel` | `LocalBasketRepositoryContract` |
| `MyOrdersViewModel` | `OrderRepositoryContract` |
| `OrderDetailViewModel` | `OrderRepositoryContract` |
| `ReturnRequestViewModel` | `OrderRepositoryContract` |

실제 API 연결 시 권장 순서:

1. `core/api`에 공통 API 클라이언트를 만든다.
2. `ApiProductRepository`, `ApiOrderRepository` 같은 실제 구현체를 추가한다.
3. 앱 시작 지점 또는 간단한 Repository provider에서 ViewModel에 실제 구현체를 주입한다.
4. 서버 오류는 ViewModel에서 고객용 문구로 변환해 화면에 표시한다.
5. 서버 상태 코드는 `OrderStatusMapper`를 통해 고객용 한글 문구로 변환한다.

## 8. 실제 API 연결 우선순위

프론트 화면 완성 후 실제 API를 붙일 때는 아래 순서가 가장 안정적이다.

### 1순위. 상품과 수확 슬롯

먼저 연결할 API:

- `GET /products?featured=true&limit=3`
- `GET /products`
- `GET /products/{product_id}`
- `GET /products/{product_id}/slots`

이유:

- 로그인 없이도 확인 가능한 공개 데이터다.
- 홈, 상품 목록, 상품 상세가 모두 이 데이터에 의존한다.
- 상품/슬롯 응답 구조가 맞아야 예약함, 예약 확인, 주문 흐름을 안정적으로 붙일 수 있다.
- MVP 사과 카테고리는 `양광`, `부사` 기준으로 맞춘다.

프론트에서 확인할 것:

- 상품 목록 필터가 `양광`, `부사`만 표시되는지
- 상품 상세의 패키지 단위와 수확 슬롯이 맞게 표시되는지
- 잔여 수량과 가격이 슬롯 기준으로 표시되는지

### 2순위. 인증과 로그인 세션

먼저 연결할 API:

- `POST /auth/customers/signup`
- `POST /auth/email/verify`
- `POST /auth/email/resend`
- `POST /auth/login`
- `GET /me`

이유:

- 예약 생성, 주문, 마이페이지, 반품은 로그인 상태가 필요하다.
- 현재 Mock 로그인 상태를 실제 토큰 기반 상태로 바꿔야 이후 API 연결이 자연스럽다.

프론트에서 확인할 것:

- 로그인 성공 시 토큰 저장 위치
- 새로고침 후 로그인 유지 여부
- 로그아웃 시 토큰 삭제
- 이메일 인증 상태 표시

### 3순위. 마이페이지와 배송지

먼저 연결할 API:

- `GET /me`
- `PATCH /me`
- `GET /me/addresses`
- `POST /me/addresses`
- `PATCH /me/addresses/{address_id}`
- `PATCH /me/addresses/{address_id}/default`

이유:

- 주문서 작성 화면에서 배송지 선택과 자동 채움이 이미 구현되어 있다.
- 배송지 API를 먼저 붙이면 주문서 작성 API 연결 전에도 화면 검증이 쉽다.
- 카카오 주소 검색은 프론트에서 주소 값을 채우고, 백엔드는 저장값만 받으면 된다.

프론트에서 확인할 것:

- 기본 배송지 선택
- 최근 사용 배송지 표시
- 배송지 별명은 선택값으로 처리
- 상세주소 미입력/입력 흐름

### 4순위. 예약과 주문 생성

먼저 연결할 API:

- `POST /reservations/preview`
- `POST /reservations`
- `POST /orders/from-reservation`

이유:

- 예약 가능 수량, 가격, 슬롯 상태 검증은 서버가 최종 판단해야 한다.
- 로컬 예약함은 서버 저장 전 단계이므로, 예약 생성 성공 시점부터 서버 데이터로 넘어가야 한다.

프론트에서 확인할 것:

- 서버가 내려준 예약 가능 여부 메시지
- 수량 부족, 마감 슬롯, 가격 변경 오류 처리
- 예약 만료 시간이 있다면 주문서 작성 화면에 표시 가능

### 5순위. 결제와 주문 조회

먼저 연결할 API:

- `POST /payments/mock-approve`
- `GET /me/orders`
- `GET /me/orders/{order_id}`
- `GET /me/orders/{order_id}/shipment`

이유:

- 결제 승인 후 주문 상태가 `PROCUREMENT_REQUESTED`로 바뀌어야 한다.
- 내 주문과 주문 상세는 상태 매핑, 배송 정보, 반품 버튼 노출 기준을 검증하는 핵심 화면이다.

프론트에서 확인할 것:

- 결제 성공 후 주문 완료 화면 이동
- 주문 목록의 상태 문구
- 주문 상세 타임라인
- 배송 중 또는 배송 완료 상태의 택배사/송장번호 표시

### 6순위. 반품

먼저 연결할 API:

- `POST /returns`
- `GET /me/returns`

이유:

- 반품은 주문과 배송 상태가 먼저 연결된 뒤 붙이는 것이 안전하다.
- 현재 프론트 기준은 배송 완료 상태에서 반품 신청 버튼을 표시한다.

프론트에서 확인할 것:

- 반품 상세 사유 10자 이상 검증
- 반품 요청 후 주문 상태가 `RETURN_REQUESTED` 또는 `반품 요청 접수`로 보이는지
- 이미 반품 요청한 주문에서 중복 신청을 막는지

### 7순위. 고객센터와 정책 링크

먼저 결정할 항목:

- 프론트 고정값 유지
- 설정 API 또는 관리자 관리값으로 분리

후보 필드:

- `support_phone`
- `support_email`
- `support_hours`
- `terms_url`
- `privacy_url`

MVP에서는 현재처럼 프론트 고정값으로도 충분하다.

## 9. 발표 또는 시연 전 체크리스트

- 데스크톱 첫 화면 확인
- 모바일 첫 화면 확인
- 홈 상품 클릭 시 상품 상세 이동 확인
- 상품 상세에서 예약함 담기 확인
- 예약함 수량 변경/선택 삭제 확인
- 예약 확인 → 주문서 작성 → 결제 → 주문 완료 흐름 확인
- 주문 상세 상태 타임라인 확인
- 반품 신청 → 반품 요청 완료 흐름 확인
- 로그인, 회원가입, 이메일 인증 이동 확인
- 마이페이지 진입 확인
- 배송지 관리/추가/수정 확인
- 회원 정보 보기/수정 확인
- 모바일 overflow 경고 확인
- `dart analyze` 통과 확인
