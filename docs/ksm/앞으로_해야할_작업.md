# 사용자 웹 앞으로 해야 할 작업

작성일: 2026-05-02

이 문서는 현재 디자인 작업 이후 `SMART_WEB` 사용자 웹에서 앞으로 진행해야 할 작업을 정리한 문서다.

## 1. 우선순위 요약

| 우선순위 | 작업 | 목적 |
|---|---|---|
| 1 | 전체 화면 직접 QA | 디자인 깨짐, 버튼 이동, 라우팅 오류 확인 |
| 2 | 더미 데이터 정리 | 화면별 데이터 일관성 확보 |
| 3 | 상태 변화 흐름 구현 | 예약, 주문, 반품 후 상태가 자연스럽게 바뀌도록 처리 |
| 4 | 백엔드 API 연결 준비 | 담당자에게 필요한 요청/응답 필드 전달 |
| 5 | 실제 API 연동 | 더미 Repository를 실제 API Repository로 교체 |
| 6 | 예외/빈 상태 처리 | 로딩, 실패, 빈 목록 화면 완성 |
| 7 | 최종 발표/시연 점검 | 시연 흐름과 화면 안정성 확인 |

## 2. 바로 확인해야 할 QA 목록

아래 화면을 데스크톱과 모바일 크기에서 모두 확인한다.

- 홈
- 상품 목록
- 상품 상세
- 담아둔 수확 상품
- 예약 전 확인
- 주문서 작성
- 결제
- 주문 완료
- 내 주문
- 주문 상세
- 반품 신청
- 반품 요청 완료
- 로그인
- 회원가입
- 이메일 인증

확인할 항목:

- AppBar의 `Harvest Slot` 클릭 시 홈으로 이동하는지
- 버튼 클릭 시 다음 화면으로 이동하는지
- 모바일에서 overflow 경고가 없는지
- 한글이 깨지지 않는지
- 고객에게 어색한 개발용 문구가 없는지
- 이미지 로딩 실패 시 화면이 비어 보이지 않는지

## 3. 더미 데이터 정리

현재는 화면 구현용 더미 데이터가 섞여 있다.

정리해야 할 항목:

- 상품명과 농장명 통일
- 수확 예정 날짜 통일
- 주문번호 통일
- 주문 상태 통일
- 결제 금액과 상품 금액 일치
- 반품 신청 대상 주문 상태를 `배송 완료` 기준으로 정리

추천 기준:

```text
대표 주문번호: ORD-20261012-008
대표 상품: 후지 사과
대표 농장: 충주 햇살농원
대표 수확 예정: 10.12-10.18
```

## 4. 상태 변화 흐름 구현

현재는 화면 이동은 되지만 실제 상태가 저장되어 바뀌는 구조는 아직 약하다.

추가로 구현하면 좋은 흐름:

- 예약함에 담기
- 예약함에서 삭제
- 예약 전 확인 후 예약 생성
- 주문서 작성 후 결제 화면 이동
- 결제 완료 후 주문 완료 이동
- 주문 완료 후 내 주문 목록에 주문 표시
- 반품 신청 후 내 주문 상태가 `반품 요청 접수`로 표시

현재는 대부분 더미 데이터 기반이므로, 최소한 로컬 상태 또는 Repository 더미 데이터에서 흐름이 이어지도록 정리하면 시연 완성도가 올라간다.

## 5. 백엔드 연결 준비

백엔드 담당자에게 전달해야 할 주요 연결 지점은 아래와 같다.

### 인증

- `POST /auth/customers/signup`
- `POST /auth/email/verify`
- `POST /auth/email/resend`
- `POST /auth/login`
- `GET /me`

필요 데이터:

- 이메일
- 비밀번호
- 이름
- 전화번호
- 인증 코드
- access token

### 상품

- `GET /products?featured=true`
- `GET /products`
- `GET /products/{product_id}`
- `GET /products/{product_id}/slots`

필요 데이터:

- 상품 ID
- 상품명
- 농장명
- 상품 이미지
- 수확 예정 범위
- 패키지 단위
- 가격
- 잔여 수량
- 예약 가능 여부

### 예약

- `POST /reservations/preview`
- `POST /reservations`

필요 데이터:

- 상품 ID
- 슬롯 ID
- 패키지 수량
- 패키지 단위 kg
- 예약 총 중량
- 예상 금액
- 예약 만료 시간

### 주문/결제

- `POST /orders/from-reservation`
- `POST /payments/mock-approve`
- `GET /me/orders`
- `GET /me/orders/{order_id}`
- `GET /me/orders/{order_id}/shipment`

필요 데이터:

- 예약 ID
- 주문 ID
- 주문번호
- 배송지
- 배송 메모
- 결제 금액
- 주문 상태
- 배송 상태
- 택배사
- 송장번호

### 반품

- `POST /returns`

필요 데이터:

- 주문 ID
- 반품 사유 코드
- 상세 사유
- 요청 금액
- 증빙 이미지 URL
- 반품 상태

## 6. 추가 기능 후보

이미 정리된 문서:

```text
docs/ksm/사용자웹_추가기능_제안.md
```

우선 고려할 기능:

- 예약 만료 카운트다운
- 기본 배송지 선택
- 잔여 수량 적음 배지
- 주문 상태 타임라인
- 반품 가능 조건 안내

현재 일부는 화면 디자인 수준으로 반영되어 있으나, 실제 데이터 연결은 아직 필요하다.

## 7. 코드 정리 작업

앞으로 정리하면 좋은 부분:

- 인증 화면 공통 레이아웃을 별도 위젯 파일로 분리
- 공통 배경 위젯 정리
- 공통 카드/요약/상태 위젯 재사용성 개선
- 더미 Repository와 실제 API Repository 분리
- 라우트 arguments 타입 정리
- 화면별 ViewModel 책임 범위 정리

추천 공통 위젯 후보:

- `BrandAppBarTitle`
- `AuthSplitLayout`
- `AuthCard`
- `AuthTextField`
- `ScreenBackground`
- `SummaryRow`
- `InfoTile`

## 8. 발표 또는 시연 전 체크리스트

- 서버 재시작 후 첫 진입 화면 확인
- 홈에서 전체 흐름 클릭 테스트
- 모바일 폭에서 overflow 경고 확인
- 이미지 로딩 실패 시 fallback 확인
- 로그인, 회원가입, 이메일 인증 이동 확인
- 상품 상세에서 예약함 담기 확인
- 예약함에서 삭제/확인 이동 확인
- 결제 완료 후 주문 상세 이동 확인
- 반품 신청 후 반품 요청 완료 이동 확인
- `flutter analyze` 통과 확인
