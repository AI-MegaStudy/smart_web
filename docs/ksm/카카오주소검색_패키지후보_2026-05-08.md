# 카카오 주소 검색 패키지 후보

작성일: 2026-05-08

이 문서는 배송지 추가/수정 화면의 `주소 찾기` 기능을 실제 카카오/다음 주소 검색과 연결하기 전에 pub.dev 후보를 조사한 기록이다.

현재는 조사 단계이며, `pubspec.yaml`에는 아직 새 패키지를 추가하지 않았다.

## 1. 작업 범위 구분

카카오 주소 검색은 프론트와 백엔드 역할이 나뉜다.

프론트 역할:

- 주소 찾기 버튼 제공
- 카카오/다음 주소 검색 UI 열기
- 사용자가 선택한 우편번호와 주소를 입력폼에 자동 채움
- 상세주소는 사용자가 직접 입력

백엔드 역할:

- 선택된 주소를 배송지로 저장
- 기본 배송지 설정
- 배송지 목록 조회
- 주문서 작성 시 선택한 배송지 저장

백엔드 저장 필드:

```text
zip_code
address
detail_address
```

## 2. 후보 1: daum_postcode_search

패키지:

```text
daum_postcode_search: ^1.1.0
```

pub.dev:

```text
https://pub.dev/packages/daum_postcode_search
```

특징:

- Daum Postcode Service 연동용 패키지
- 자체 위젯을 강제하지 않고, 로컬 서버와 HTML assets를 제공한다.
- WebView 구현은 프로젝트에서 선택해야 한다.
- `webview_flutter` 또는 `flutter_inappwebview` 같은 WebView 패키지와 함께 사용하는 구조다.
- 주소 결과에서 `zonecode`, `address`, `roadAddress`, `jibunAddress` 등을 받을 수 있다.

장점:

- 비교적 최근 업데이트된 후보
- WebView 패키지를 강제하지 않아 구조가 유연하다.
- 콜백 파서와 HTML assets가 제공되어 직접 JS를 모두 작성하는 것보다 부담이 적다.

단점:

- Flutter Web에서 바로 간단히 붙이는 형태라기보다 WebView/로컬 서버 구조 이해가 필요하다.
- 모바일 앱, 데스크톱, 웹 환경별 동작 확인이 필요하다.
- 프로젝트가 Flutter Web 중심이면 실제 브라우저 팝업/iframe 방식이 더 단순할 수 있다.

적용 판단:

- 앱까지 같이 고려한다면 후보로 볼 수 있다.
- 현재 사용자 웹만 빠르게 처리하려면 다소 무거울 수 있다.

## 3. 후보 2: kpostal_web

패키지:

```text
kpostal_web: ^0.0.6
```

pub.dev:

```text
https://pub.dev/packages/kpostal_web
```

특징:

- Flutter Web에서 카카오 주소 검색을 쓰기 위한 패키지다.
- `KakaoAddressWidget`을 제공한다.
- 기존 `kpostal`이 웹을 지원하지 않는 점을 보완하기 위해 만들어진 패키지다.

장점:

- 이름 그대로 Flutter Web 목적에 맞는 후보
- 위젯 형태로 사용할 수 있어 화면에 붙이기 쉽다.
- 주소 선택 완료 콜백을 받을 수 있다.

단점:

- 다운로드 수와 사용 사례가 많지 않다.
- 패키지 분석 상태가 완전하지 않은 것으로 보인다.
- 유지보수 상태를 더 확인해야 한다.

적용 판단:

- 사용자 웹에서만 주소 검색을 붙인다면 가장 단순한 후보가 될 수 있다.
- 다만 안정성 검토 후 적용하는 것이 좋다.

## 4. 후보 3: kakao_address_web

패키지:

```text
kakao_address_web: ^0.0.2
```

pub.dev:

```text
https://pub.dev/packages/kakao_address_web
```

특징:

- Flutter Web에서 카카오 주소 검색 팝업 형태를 제공하는 패키지다.
- 우편번호, 주소, 건물명 등을 콜백으로 받을 수 있다.

장점:

- 팝업형 주소 검색을 목표로 해서 현재 화면 요구와 잘 맞는다.
- 사용 방식이 비교적 단순해 보인다.

단점:

- 버전이 낮고 사용량이 적다.
- 유지보수와 실제 호환성 검토가 필요하다.
- `kpostal_web`에서 영감을 받은 형태라 두 패키지 중 어느 쪽이 더 안정적인지 테스트가 필요하다.

적용 판단:

- 빠른 프로토타입에는 가능성이 있다.
- 실제 적용 전 간단한 브랜치에서 동작 테스트가 필요하다.

## 5. 현재 추천

현재 프로젝트는 사용자 웹 프론트 시연 완성도가 우선이므로, 바로 패키지를 추가하기보다는 아래 순서를 추천한다.

1. 지금은 `주소 찾기` 버튼을 유지하고 안내 팝업만 둔다.
2. 카카오 주소 검색을 실제로 붙이기로 결정되면 `kpostal_web`을 1차 후보로 테스트한다.
3. `kpostal_web`이 원하는 팝업/화면 흐름과 맞지 않으면 `kakao_address_web`을 비교한다.
4. 앱까지 함께 고려해야 한다면 `daum_postcode_search`를 검토한다.

추천 우선순위:

```text
1순위: kpostal_web
2순위: kakao_address_web
3순위: daum_postcode_search
```

## 5-1. 팀장 추천 패키지 반영

팀장이 실제로 사용해본 패키지로 `kpostal_plus`를 추천했고, 현재 프로젝트에는 이 패키지를 우선 적용했다.

적용 패키지:

```text
kpostal_plus: ^1.0.3
```

pub.dev:

```text
https://pub.dev/packages/kpostal_plus
```

적용 위치:

```text
lib/presentation/views/mypage/address_manage_page.dart
lib/presentation/views/checkout/checkout_page.dart
```

적용 내용:

- 배송지 추가/수정 화면의 `주소 찾기` 버튼을 실제 주소 검색 화면으로 연결했다.
- `KpostalPlusView`를 열고, 선택 결과를 `Kpostal`로 받는다.
- 선택된 주소의 `postCode`를 우편번호 필드에 채운다.
- 선택된 주소의 `userSelectedAddress`를 주소 필드에 채운다.
- 상세주소는 기존처럼 사용자가 직접 입력한다.
- 주문서 작성 화면의 `주소 검색` 버튼도 같은 주소 검색 화면으로 연결했다.
- 주문서 작성 화면은 현재 주소 필드가 하나이므로, 선택 결과를 `(우편번호) 주소` 형태로 채운다.

현재 기준 추천 우선순위는 아래처럼 변경한다.

```text
1순위: kpostal_plus
2순위: kpostal_web
3순위: kakao_address_web
4순위: daum_postcode_search
```

검증:

```text
flutter pub get
dart format
dart analyze
```

정적 분석 결과:

```text
No issues found!
```

사용자 확인 결과:

- 배송지 추가/수정 화면에서 주소 검색 화면이 정상적으로 열리는 것을 확인했다.
- 주소 선택 후 우편번호와 주소가 입력폼에 정상 반영되는 것을 확인했다.
- 주문서 작성 화면에서도 주소 검색 화면이 정상적으로 열리고, 선택 주소가 주소 필드에 반영되도록 추가 적용했다.

## 6. 적용 전 확인해야 할 것

- Flutter Web 빌드에서 정상 동작하는지
- 주소 검색 팝업이 브라우저 팝업 차단에 걸리지 않는지
- 선택 후 `zip_code`, `address`를 정상적으로 받을 수 있는지
- 상세주소는 기존 입력폼과 자연스럽게 연결되는지
- 모바일 브라우저 크기에서도 검색 UI가 깨지지 않는지
- 패키지 추가 시 `dart format`, `dart analyze` 통과 여부
