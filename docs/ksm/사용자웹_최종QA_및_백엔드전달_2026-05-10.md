# 사용자웹 최종 QA 및 백엔드 전달 정리

작성일: 2026-05-10

이 문서는 사용자 웹에서 실제 API 연결 후 확인한 내용, 현재 남은 확인 항목, 백엔드 담당자/팀장님에게 전달해야 할 내용을 한곳에 모은 최신 정리 문서다.

## 0. 2026-05-11 시연 마감 최신 기준

- 고객 화면 품종 기준은 `부사 사과`, `신고 사과`이다.
- 농장명 표시는 `문경 햇살 농장` 하나로 고정한다.
- 과거 `양광/부사` 기준은 현재 고객 화면 기준이 아니다.
- 홈, 상품, 예약함, 주문 상세에서 `양광`, `신고 배`, `Harvest Demo Farm`, `QA 고객`, `customer@test.com`, `Mock 결제`, `0원` 같은 시연용 표현이 보이지 않도록 정리했다.
- 결제 대기 주문은 결제 정보 영역에서 승인금액 `0원` 대신 결제 예정 금액을 표시한다.
- 반품 요청은 증빙 이미지 여러 장 선택과 요청 금액 직접 입력을 지원한다.
- 이용약관과 개인정보처리방침은 고객센터 안내가 아니라 본문 모달로 표시한다.
- ML 예측 API는 OWNER/백엔드 범위이며 고객 예약 웹에는 예측값을 직접 표시하지 않는다.

## 1. 이메일 인증 최신 상태

팀장 전달 기준으로 이메일 인증 DB 문제는 해결됐고, 실제 이메일 인증 흐름까지 정상 확인한 상태다.

해결된 항목:

- `email_verifications.attempt_count` 컬럼 추가
- `email_verifications.updated_at` 컬럼 추가
- 가입 전 인증을 위해 `email_verifications.account_id`를 `NULL` 허용으로 변경
- 해시 저장을 위해 `email_verifications.verification_code`를 `VARCHAR(255)`로 변경

현재 백엔드 코드 기준:

- `EmailVerification` 모델은 `TimestampMixin`을 다시 사용한다.
- 인증 코드 실패 시 `attempt_count`를 증가시키고 최대 실패 횟수 초과 시 제한한다.
- 가입 전 인증 성공 후 회원가입이 완료되면 해당 인증 레코드의 `account_id`를 새 계정과 연결한다.
- 백엔드 환경변수는 프로젝트 루트 `.env`를 기준으로 읽는다.

현재 프론트 상태:

- 회원가입 화면에서 이메일 인증 단계로 이동하는 흐름은 구현되어 있다.
- 고객 화면에 테스트 인증 코드를 직접 노출하는 문구는 제거했다.
- 이메일 발송, 6자리 인증 코드 확인, 고객 회원가입, 로그인 흐름을 확인했다.

## 2. 주문/결제/마이페이지 QA

사용자가 직접 확인한 항목:

- 새 계정 기준 이메일 인증 메일 발송
- 메일로 받은 6자리 인증 코드 검증
- 이메일 인증 후 고객 회원가입 완료
- 회원가입 후 로그인 성공
- 로그인 후 `/me` 기준으로 마이페이지 이름, 이메일, 전화번호 표시
- 회원정보 수정 후 새로고침 없이 회원정보 보기 화면에 즉시 반영
- 전화번호 `01033334444` 형식을 `010-3333-4444`처럼 표시
- 배송지 관리, 배송지 저장, 주문서 작성 화면의 배송지 반영
- 상품 상세에서 예약함 담기
- 예약함 담기 후 바로 이동하지 않고 선택 팝업 표시
- 예약함에서 예약 진행하기
- 예약 확인 화면 금액/수량 표시
- `POST /api/v1/reservations/preview` 호출 확인
- `POST /api/v1/reservations` 호출 및 DB 저장 확인
- 주문서 작성 화면 이동 확인
- 주문서의 결제 화면 이동 버튼에서 주문 생성 API 호출
- 결제 승인 API 호출
- 주문 목록과 주문 상세에서 실제 주문 정보 표시
- 주문 상세 결제 정보 문구 정리
- 반품 내역 빈 상태 화면 확인

추가로 한 번 더 보면 좋은 최종 QA:

- 새 계정으로 예약함 → 주문서 → 결제 → 주문목록 → 주문상세 전체 흐름 재확인
- 새로고침 후 로그인 유지
- 배송지 수정 후 주문서 작성 화면 반영
- 회원정보 수정 후 마이페이지와 회원정보 보기 화면 반영

코드 연결 기준으로 회원가입, 로그인, 상품 상세, 예약함, 예약 확인, 주문서 작성, 결제, 주문 완료, 주문 목록, 주문 상세 라우트와 API 연결은 확인했다.

## 3. 임시로 바꾼 백엔드 코드

사용자 웹 연동을 위해 임시 수정한 백엔드 영역:

- `backend/app/models/account.py`
- `backend/app/services/email_verification_service.py`
- `backend/app/schemas/auth_schema.py`
- `backend/app/services/auth_service.py`
- `backend/app/routers/auth_router.py`
- `backend/app/schemas/address_schema.py`
- `backend/app/services/address_service.py`
- `backend/app/routers/address_router.py`
- `backend/app/routers/router.py`

임시 수정 및 최신 반영 내용:

- DB 보정 전에는 `email_verifications.attempt_count`, `updated_at` 참조를 임시 제거했었다.
- DB 보정 후에는 `attempt_count`와 `updated_at` 기준으로 백엔드 모델과 검증 로직을 다시 연결한다.
- 회원정보 수정용 `PUT /api/v1/me` 추가
- 배송지 관리용 `/api/v1/me/addresses` 계열 API 추가

이메일 인증 관련 임시 수정은 DB 보정 완료 기준으로 복구한다.

## 5. 배송/반품 검증 보류

배송과 반품은 사용자 웹 코드 연결은 되어 있으나 최종 성공 검증은 아직 보류한다.

보류 이유:

- 배송 정보는 농가/관리자 앱에서 배송 생성 또는 상태 변경이 완료되어야 실제 값으로 확인할 수 있다.
- 반품 신청은 백엔드 정책상 배송 완료 주문에서만 가능하므로, 농가/관리자 앱에서 주문을 배송 완료 상태까지 넘기는 흐름이 먼저 필요하다.

추후 확인할 항목:

- 주문 상세에서 택배사/운송장번호 표시
- 주문 상태가 배송 완료가 되었을 때 반품 신청 버튼 노출
- 반품 신청 후 `return_requests` 저장
- 내 반품 목록에 반품 요청 내역 표시
- 중복 반품 신청 방지

## 6. 문구 점검

현재 고객 화면에서 제거하거나 주의할 표현:

- `Mock`은 고객 화면에 직접 노출하지 않는다. 결제 화면은 `카드 결제` 또는 `카드 간편결제`로 표시한다.
- `더미`, `테스트 인증 코드`, `API`, `DB`, `database integrity error` 같은 개발자용 표현은 고객 화면에 노출하지 않는다.
- 이메일 인증 실패나 서버 오류는 고객용 안내 문구로 변환해서 표시한다.

2026-05-10 기준 반영:

- 이메일 인증 화면의 `테스트 인증 코드` 표시 제거
- 결제 정보 영역은 결제 수단, 결제 금액, 결제 상태 중심으로 정리
- 주문/배송/반품 상태는 고객용 한글 문구로 표시
- 고객 화면 표시 문자열에서 `Mock`, `더미`, `API`, `DB`, `테스트 코드`, `database integrity error` 직접 노출 여부를 검색했고 노출 문자열은 발견되지 않았다.
- `MockPaymentPage`, `MockAuthSession` 같은 이름은 내부 코드명이며 고객 화면 문구가 아니다.

## 7. 남은 작업

우선순위:

1. 새 계정 기준 예약함 → 주문서 → 결제 → 주문목록 → 주문상세 전체 흐름 최종 클릭 QA
2. 농가/관리자 앱 배송 상태 연결 후 배송/반품 최종 QA
3. 최종 발표 전 고객 화면 문구 재점검
4. 전체 `dart analyze` 통과 확인
