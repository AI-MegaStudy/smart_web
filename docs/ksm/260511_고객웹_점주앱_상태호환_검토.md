# 고객웹-점주앱 상태 호환 검토

작성일: 2026-05-11

## 검토 범위

- 참고 문서: `docs/ksm`
- 고객웹 참고 저장소: `C:/Users/ose/smart_web`
- 점주앱 저장소: `C:/Users/ose/SMART_APP2`
- 검토 방식: 문서와 코드 읽기 전용 확인

## 결론

고객웹과 점주앱의 백엔드 상태 코드는 거의 동일하게 맞춰져 있다. 예약 생성, 주문 생성, 결제 승인, 농가 확인, 배송, 반품 요청, 환불까지의 기본 흐름은 같은 상태 코드를 공유한다.

기본 상태 흐름은 아래처럼 이어진다.

```text
PAYMENT_PENDING
-> PROCUREMENT_REQUESTED
-> PROCUREMENT_APPROVED / PROCUREMENT_PARTIAL_APPROVED / PROCUREMENT_REJECTED
-> SHIPPED
-> DELIVERED
-> RETURN_REQUESTED
-> REFUNDED / REJECTED
```

다만 시연 기준으로는 일부 화면 라벨과 진행 UI가 상태 코드를 완전히 세분화해서 표현하지 못하는 구간이 있다. 특히 `PROCUREMENT_APPROVED`, `PROCUREMENT_PARTIAL_APPROVED`, `PROCUREMENT_REJECTED`, `READY_TO_SHIP`, `QUALITY_CHECKING`, `REFUNDED` 단계는 고객웹과 점주앱에서 보이는 문구나 화면 흐름이 약간 다를 수 있다.

또한 품종/농장명은 현재 설명과 코드가 다르다. 요청 내용은 `부사/신고`, `문경 햇살 농장` 기준이지만, 실제 코드상 고객웹과 점주앱 모두 상당 부분 `양광/부사` 기준으로 되어 있다.

## 1. 고객웹 예약/주문 데이터의 점주앱 노출

정상적으로 이어질 가능성이 높다.

고객웹에서 주문을 만들면 서버는 먼저 `PAYMENT_PENDING` 주문을 생성한다. 이후 고객웹 결제 승인 API가 호출되면 결제는 `APPROVED`, 주문은 `PROCUREMENT_REQUESTED`, 발주는 `REQUESTED`로 생성된다.

점주앱은 아래 API를 통해 이 데이터를 읽는다.

- `/owner/orders`
- `/owner/reservations`
- `/owner/procurements`

따라서 고객웹에서 결제까지 완료된 주문은 점주앱의 주문/발주 흐름에서 확인 가능하다.

주의할 점은 점주앱 주문 목록의 단순 상태 라벨이다. 점주앱 `OwnerOrderRecord`는 `PAID` 또는 `PROCUREMENT_REQUESTED`를 모두 `결제 완료`로 축약해서 보여준다. 실제 발주 처리 상태는 발주 화면에서 확인하는 구조에 가깝다.

## 2. 점주앱 승인/배송 상태의 고객웹 반영

대체로 반영된다.

점주앱에서 발주 승인/부분승인/거절을 처리하면 서버 주문 상태는 아래처럼 바뀐다.

| 점주앱 처리 | 서버 주문 상태 | 고객웹 표시 |
|---|---|---|
| 승인 | `PROCUREMENT_APPROVED` | 농가 승인 완료 |
| 부분승인 | `PROCUREMENT_PARTIAL_APPROVED` | 일부 수량 확인 필요 |
| 거절 | `PROCUREMENT_REJECTED` | 농가 승인 불가 |

배송 등록과 배송 상태 변경도 고객웹 주문 상세에 반영된다.

| 점주앱 처리 | 서버 배송 상태 | 서버 주문 상태 | 고객웹 표시 |
|---|---|---|---|
| 배송 등록 | `SHIPPED` | `SHIPPED` | 배송 중 |
| 배송 완료 처리 | `DELIVERED` | `DELIVERED` | 배송 완료 |

2026-05-12 고객웹 반영:

- 고객웹 주문 상세 진행 타임라인이 서버 원본 `order_status`, `return_status`를 함께 보도록 수정했다.
- `PROCUREMENT_APPROVED`는 `농가 승인 완료`로 현재 단계 표시한다.
- `PROCUREMENT_PARTIAL_APPROVED`는 `일부 수량 확인 필요`로 현재 단계 표시한다.
- `PROCUREMENT_REJECTED`는 `농가 승인 불가`로 표시하고 배송 단계는 숨긴다.
- `QUALITY_CHECKING`은 `선별 중`, `READY_TO_SHIP`은 `발송 준비`로 표시한다.

## 3. 고객웹 반품 요청의 점주앱 확인

가능하다.

고객웹에서 반품 요청을 생성하면 서버는 아래처럼 처리한다.

- `return_status = REQUESTED`
- `order_status = RETURN_REQUESTED`

점주앱은 `/owner/returns`를 조회하고, `REQUESTED` 상태를 `접수`로 변환해 반품 관리 화면에 보여준다. 따라서 고객웹 반품 요청은 점주앱에서 확인 가능하다.

## 4. 점주앱 반품 확인/환불 후 고객웹 상태

승인/환불 흐름은 이어진다.

점주앱에서 반품을 승인하면 서버는 별도의 중간 `APPROVED` 표시로 오래 머물지 않고 곧바로 환불 완료 상태로 처리한다.

- `return_status = REFUNDED`
- `order_status = REFUNDED`
- `payment_status = REFUNDED`
- `refund_status = COMPLETED`

고객웹은 `REFUNDED`를 `환불 완료`로 표시한다. 따라서 고객 입장에서는 반품 요청 후 환불 완료로 자연스럽게 이어진다.

주의할 점은 점주앱 표시 방식이다. 점주앱 반품 현황은 `APPROVED`와 `REFUNDED`를 모두 `승인`으로 묶어 보여준다. 그래서 고객웹은 `환불 완료`, 점주앱은 `승인`으로 보일 수 있다.

반품 거절은 서버에서 `return_status = REJECTED`로 바뀐다. 고객웹 주문 상세는 `return_status`를 함께 확인해서 `반품 반려`로 표시하도록 보완했다.

2026-05-12 고객웹 반영:

- `return_status = REQUESTED`는 `반품 요청 접수`로 표시한다.
- `return_status = APPROVED`는 `반품 승인`으로 표시한다.
- `return_status = REJECTED`는 `반품 반려`로 표시한다.
- `order_status = REFUNDED` 또는 환불 완료 상태는 `환불 완료`로 표시한다.

## 5. 품종/농장명 호환성

현재 가장 큰 불일치 지점이다.

요청 기준:

- 품종: `부사`, `신고`
- 농장명: `문경 햇살 농장`

실제 코드 기준:

- 고객웹 더미 데이터: `양광`, `부사`, `문경 햇살 농장`
- 고객웹 API 매핑: `shingo`, `신고`를 `양광`으로 변환
- 점주앱 품종 목록: `양광`, `부사`
- 점주앱 더미 농장명: `청주 햇살농원`

즉, 현재 코드 그대로라면 `부사/신고` 시연보다 `부사/양광` 시연이 더 자연스럽다. `신고`를 그대로 쓰려면 고객웹의 품종 매핑과 점주앱 품종 목록/더미 데이터가 함께 맞아야 한다.

## 고객웹에는 있는데 점주앱 처리나 화면 흐름이 애매한 단계

| 상태 코드 | 고객웹 표시 | 점주앱/서버 처리 | 검토 의견 |
|---|---|---|---|
| `PAYMENT_PENDING` | 결제 대기 | 주문 목록에서 `주문 완료`로 축약 가능 | 결제 전 주문은 점주앱에서 발주 처리 대상은 아님 |
| `PROCUREMENT_REQUESTED` | 농가 확인 중 | 발주 승인 대기 | 호환 양호 |
| `PROCUREMENT_APPROVED` | 농가 승인 완료 | 발주 승인 | 주문 상세 타임라인 반영 완료 |
| `PROCUREMENT_PARTIAL_APPROVED` | 일부 수량 확인 필요 | 부분승인 | 주문 상세 타임라인 반영 완료, 후속 조정 UX는 추후 과제 |
| `PROCUREMENT_REJECTED` | 농가 승인 불가 | 거절 | 주문 상세 타임라인 반영 완료, 취소/환불/재주문 흐름은 추후 과제 |
| `QUALITY_CHECKING` | 선별 중 | 코드에는 있으나 주요 전이에서 뚜렷하지 않음 | 주문 상세 타임라인 표시 반영 완료 |
| `READY_TO_SHIP` | 발송 준비 | 코드에는 있으나 현재 배송 등록은 바로 `SHIPPED` | 주문 상세 타임라인 표시 반영 완료 |
| `SHIPPED` | 배송 중 | 배송 등록/상태 변경 가능 | 호환 양호 |
| `DELIVERED` | 배송 완료 | 배송 완료 처리 가능 | 호환 양호 |
| `RETURN_REQUESTED` | 반품 요청 접수 | 반품 관리에서 접수로 표시 | 호환 양호 |
| `REFUNDED` | 환불 완료 | 점주앱은 반품 현황에서 승인으로 축약 | 문구 차이 있음 |

## 최종 의견

상태 코드 호환성은 전반적으로 맞다. 고객웹에서 생성된 주문과 반품 요청은 점주앱에서 확인 가능하고, 점주앱의 승인/배송/반품 처리도 고객웹에 반영되는 구조다.

시연 전에 우선 확인할 부분은 두 가지다.

1. 품종 기준을 `부사/신고`로 갈지, 현재 코드와 맞는 `부사/양광`으로 갈지 결정해야 한다.
2. 고객웹 주문 상세 타임라인의 `농가 승인 완료`, `부분승인`, `거절`, `환불 완료` 표시는 2026-05-12 기준 1차 반영했다. 남은 과제는 부분승인/거절 이후 고객 안내와 취소/재주문 UX다.

현재 코드 그대로라면 시연 문구는 `부사/양광`, 농장명은 고객웹 `문경 햇살 농장`과 점주앱 더미 `청주 햇살농원`이 섞여 보일 수 있다는 점을 주의해야 한다.
