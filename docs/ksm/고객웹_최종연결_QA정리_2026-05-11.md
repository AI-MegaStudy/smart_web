# 고객웹 최종 연결 QA 정리 - 2026-05-11

## 1. 이번 정리 기준

고객 웹은 `부사 사과`, `양광 사과` 기준으로 정리한다.

고객 화면에는 아래 표현을 직접 노출하지 않는다.

- QA
- 더미
- Mock
- DB
- API
- 테스트 계정
- Harvest Demo Farm
- customer@test.com
- database integrity error

코드 내부 클래스명, API 경로명, 파일명에 남아 있는 `MockAuthSession`, `MockPaymentPage`, `mock-approve`는 고객 화면 문구가 아니므로 이번 정리 대상에서 제외한다.

## 2. 상품/농장 연결

백엔드 상품 응답에 농장 상세 필드를 추가했다.

- `farm_address`
- `farm_description`
- `delivery_policy`
- `return_policy`
- 기존 연결 유지: `farm_name`, `farm_region`, `farm_image_url`

프론트 `ProductModel`에는 아래 필드를 추가 또는 반영했다.

- `productDescription`
- `farmRegion`
- `farmAddress`
- `farmImageUrl`
- `farmDescription`
- `deliveryPolicy`
- `returnPolicy`

상품 상세 화면에서는 `products.product_description`과 `farms.farm_description`을 함께 보여준다.

농장 이미지 정책:

- `farm_image_url` 값이 있을 때만 농장 이미지를 표시한다.
- 값이 없거나 이미지 로드에 실패하면 대체 박스를 보여주지 않고 영역을 숨긴다.

## 3. 상품 표시 기준 변경

기존 `신고 사과` 기준을 `양광 사과` 기준으로 변경했다.

반영 위치:

- 상품 API 매핑
- 로컬 상품/예약함 더미 기준
- 주문 상품명 보정
- 홈 품종 소개와 배지
- DB 상품명/품종 일부 값
- QA 데이터 생성 스크립트

DB 조회 기준 현재 `products`에서 `신고` 상품은 남아 있지 않다.

## 4. 홈 대표 상품 이동

홈 히어로의 `대표 상품` 버튼은 상품 목록으로 이동하지 않고, 예약 가능한 사과 상품 중 잔여 수량이 가장 많은 상품의 상세 화면으로 이동한다.

대표 상품을 찾지 못하면 기존처럼 상품 목록으로 이동한다.

## 5. 예약함 정리

예약함에 남아 있던 시연용 기본 상품 2개를 제거했다.

결제 완료 후에는 로컬 예약함을 비우도록 정리했다.

확인 기준:

- 새로고침 후 예약함이 비어 있으면 정상
- 상품 상세에서 예약함 담기 후에만 예약함에 표시
- 결제 완료 후 예약함이 다시 비어 있으면 정상

## 6. 주문/결제 연결

주문 상세 화면에 아래 필드를 추가로 연결했다.

- `orders.receiver_phone`
- `orders.delivery_memo`
- `orders.total_amount`

결제 정보에는 아래 필드를 추가로 연결했다.

- `payments.payment_method`
- `payments.approved_at`

화면 표시 기준:

- 주문 상세에서 연락처와 배송 요청사항을 확인할 수 있다.
- 결제 금액은 프론트 합산값보다 서버 확정 금액을 우선한다.
- 결제 일시가 있으면 결제 정보에 표시한다.

## 7. 배송/반품 연결

배송 Repository에는 아래 필드 파싱을 추가했다.

- `shipments.shipped_package_count`
- `shipments.shipped_kg`

다만 주문 요약의 상품 칩에 박스 수와 중량이 이미 표시되므로, 주문 상세 화면에는 중복 표시하지 않는다.

반품 내역에는 아래 필드를 미리 연결했다.

- `return_requests.approved_amount`
- `return_requests.decision_reason`
- `return_requests.decided_at`

값이 있을 때만 표시한다.

## 8. 내부/QA 문구 정리

수확 슬롯 카드에서 `QA 수확 슬롯 308` 같은 형태의 안내 문구는 화면에서 숨긴다.

반품 증빙 이미지의 임시 저장 경로는 `/mock/returns/...` 대신 `/returns/evidence/...`로 변경했다.

## 9. DB 조회 결과: 수정하지 않고 리스트업만 한 후보

이번 단계에서는 DB를 수정하지 않고 후보만 조회했다.

상품 후보:

- `product_id 1` 부사 사과: `image_url` 비어 있음
- `product_id 2` 양광 사과: `image_url` 비어 있음
- `product_id 5` 양광 사과: `image_url` 비어 있음
- `product_id 6` 부사 사과: `fruit_type = 사과`, `image_url` 비어 있음

수확 슬롯 후보:

- `slot_id 2`: `slot_status = CLOSE`, `customer_notice = 마감된 테스트 슬롯입니다.`
- `slot_id 4 ~ 27`: `customer_notice = QA 수확 슬롯 ...`

농장 후보:

- 조회 기준 문제 후보 없음
- `farm_image_url`이 비어 있는 것은 현재 정책상 문제로 보지 않는다.

주문/반품 번호 후보:

- `QA-ORD`, `QA-RET` 후보 0건

## 10. 검증

수정 파일 중심으로 `dart analyze`를 여러 차례 실행했고 통과했다.

확인한 주요 묶음:

- 상품/홈/예약함/주문 상세
- 결제/배송/반품 Repository
- 수확 슬롯 카드
- 반품 요청 ViewModel

`dart format`은 적용됐으나 Windows 환경에서 Dart telemetry 파일 권한 경고가 반복적으로 발생했다.

권한 경고는 포맷 적용 후 telemetry 세션 파일의 수정 시간 갱신에서 발생한 것이며, 코드 포맷 자체는 적용됐다.
