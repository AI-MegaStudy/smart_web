# 백엔드 전달 요약

작성일: 2026-05-08

이 문서는 현재 사용자 웹 프론트 화면을 실제 API와 연결할 때 백엔드 담당자에게 전달할 핵심 내용을 정리한 문서다.

## 1. 홈 추천 상품

홈의 `이번 주 오픈 수확 슬롯`은 최신 3개만 노출하는 방향으로 정리했다.

권장 API:

```text
GET /api/v1/products?featured=true&limit=3
```

필요 응답 필드:

```text
product_id
product_name
farm_name
thumbnail_url
variety_name
harvest_start
harvest_end
package_unit_kg
available_package_count
price
slot_status
```

참고:

- 상품 목록 페이지에는 이미 품종 필터가 있으므로 홈에는 품종 필터를 중복 추가하지 않는다.
- 홈은 고객에게 대표 상품 3개만 빠르게 보여주는 역할이다.

## 2. 배송지 목록과 주문서 작성 연결

주문서 작성 화면은 배송지 목록을 받아 카드로 표시하고, 선택한 배송지의 값을 주문서 입력폼에 자동 채움한다.

권장 API:

```text
GET /api/v1/me/addresses
```

필요 응답 필드:

```text
address_id
address_label
receiver_name
receiver_phone
zip_code
address
detail_address
delivery_memo
is_default
is_recent
```

프론트 표시 기준:

- `address_label`이 있으면 `내집`, `부모님댁` 같은 배송지 별명으로 표시한다.
- `address_label`이 없으면 별명 줄은 표시하지 않는다.
- `is_default = true`이면 `기본 배송지` 배지를 표시한다.
- `is_recent = true`이면 `최근 사용` 배지를 표시한다.
- 주문서 작성 진입 시 기본 배송지를 우선 선택값으로 사용할 수 있다.

## 3. 주문 상태 문구 매핑

고객 화면에는 서버 상태 코드를 직접 노출하지 않고 한글 문구로 표시한다.

| 서버 상태 | 고객 표시 |
|---|---|
| `PAYMENT_PENDING` | 결제 대기 |
| `PAID` | 결제 완료 |
| `PROCUREMENT_REQUESTED` | 농가 확인 중 |
| `PROCUREMENT_APPROVED` | 농가 승인 완료 |
| `PROCUREMENT_PARTIAL_APPROVED` | 일부 수량 확인 필요 |
| `PROCUREMENT_REJECTED` | 농가 승인 불가 |
| `QUALITY_CHECKING` | 선별 중 |
| `READY_TO_SHIP` | 발송 준비 |
| `SHIPPED` | 배송 중 |
| `DELIVERED` | 배송 완료 |
| `RETURN_REQUESTED` | 반품 요청 접수 |
| `REFUNDED` | 환불 완료 |

주문 상세 화면의 상태 타임라인 표시를 위해 아래 시간이 있으면 좋다.

```text
paid_at
procurement_approved_at
quality_checked_at
shipped_at
delivered_at
return_requested_at
```

## 4. 배송 정보 표시

배송 중 또는 배송 완료 상태에서는 택배사와 송장번호를 표시한다.

권장 API:

```text
GET /api/v1/me/orders/{order_id}/shipment
```

필요 응답 필드:

```text
carrier_name
tracking_number
shipment_status
shipped_at
delivered_at
```

프론트 표시 기준:

- `SHIPPED`, `DELIVERED`, `RETURN_REQUESTED` 상태에서는 택배사와 송장번호를 표시한다.
- 아직 배송 전이면 `배송 준비 중`, `송장번호 준비 중` 같은 고객 문구로 표시한다.

## 5. 반품 신청

현재 프론트 기준은 배송 완료 상태에서 반품 신청 버튼을 노출한다.

권장 API:

```text
POST /api/v1/returns
```

요청 필드 예시:

```json
{
  "order_id": 8,
  "reason_code": "DAMAGED",
  "reason_detail": "받은 사과 일부가 파손되어 반품을 요청합니다.",
  "request_amount": 39000,
  "evidence_image_urls": [
    "https://example.com/evidence-1.jpg"
  ]
}
```

프론트 검증 기준:

- 반품 상세 사유는 10자 이상 입력해야 한다.
- 부분 금액 선택은 제거하고 전체 반품 중심으로 정리했다.
- 반품 신청 후 주문 상태는 `RETURN_REQUESTED` 또는 고객 표시 `반품 요청 접수`로 보여준다.

## 6. 결제

현재 결제 화면은 시연용 Mock 결제 기준이다.

권장 API:

```text
POST /api/v1/payments/mock-approve
```

요청 필드 예시:

```json
{
  "order_id": 8,
  "idempotency_key": "mock-pay-20260508-0001"
}
```

프론트 기준:

- `카드 간편결제`만 시연 결제로 진행한다.
- `무통장 입금`, `카카오페이`, `네이버페이`는 준비중 팝업을 표시한다.
- 실제 결제 연동 전까지 고객 화면에는 개발자용 `Mock` 표현을 직접 노출하지 않는다.

## 7. 카카오 주소 검색

카카오 주소 검색은 프론트에서 주소 검색 UI를 띄우고, 선택된 주소를 입력폼에 채우는 작업이다.

백엔드에서는 최종 저장값만 받으면 된다.

필요 저장 필드:

```text
zip_code
address
detail_address
```

적용 전 확인:

- Flutter Web 지원 패키지 또는 JS 연동 방식 검토가 필요하다.
- 프로젝트 규칙상 새 패키지는 pub.dev 확인 후 사용자 승인 뒤 추가한다.
