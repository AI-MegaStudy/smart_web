# 백엔드 전달 요약

작성일: 2026-05-08

이 문서는 현재 사용자 웹 프론트 화면을 실제 API와 연결할 때 백엔드 담당자에게 전달할 핵심 내용을 정리한 문서다.

## 1. 홈 추천 상품

홈의 `이번 주 오픈 수확 슬롯`은 최신 3개만 노출하는 방향으로 정리했다.

사과 카테고리는 팀장 피드백 기준으로 MVP에서 아래 두 가지를 우선 사용한다.

```text
양광
부사
```

권장 API:

```text
GET /api/v1/products?featured=true&limit=3
```

필요 응답 필드:

```text
product_id
product_name
farm_name
thumbnail_url
variety_name
harvest_start
harvest_end
package_unit_kg
available_package_count
price
slot_status
```

참고:

- 상품 목록 페이지에는 이미 품종 필터가 있으므로 홈에는 품종 필터를 중복 추가하지 않는다.
- 홈은 고객에게 대표 상품 3개만 빠르게 보여주는 역할이다.
- 상품 API의 `variety_name` 또는 카테고리 필드는 우선 `양광`, `부사` 값을 기준으로 맞추면 된다.
- 프론트 더미 데이터도 `양광`, `부사` 두 카테고리로 정리되어 있다.

## 2. 배송지 목록과 주문서 작성 연결

주문서 작성 화면은 배송지 목록을 받아 카드로 표시하고, 선택한 배송지의 값을 주문서 입력폼에 자동 채움한다.

권장 API:

```text
GET /api/v1/me/addresses
```

필요 응답 필드:

```text
address_id
address_label
receiver_name
receiver_phone
zip_code
address
detail_address
delivery_memo
is_default
is_recent
```

프론트 표시 기준:

- `address_label`이 있으면 `내집`, `부모님댁` 같은 배송지 별명으로 표시한다.
- `address_label`이 없으면 별명 줄은 표시하지 않는다.
- `is_default = true`이면 `기본 배송지` 배지를 표시한다.
- `is_recent = true`이면 `최근 사용` 배지를 표시한다.
- 주문서 작성 진입 시 기본 배송지를 우선 선택값으로 사용할 수 있다.

## 3. 주문 상태 문구 매핑

고객 화면에는 서버 상태 코드를 직접 노출하지 않고 한글 문구로 표시한다.

| 서버 상태 | 고객 표시 |
|---|---|
| `PAYMENT_PENDING` | 결제 대기 |
| `PAID` | 결제 완료 |
| `PROCUREMENT_REQUESTED` | 농가 확인 중 |
| `PROCUREMENT_APPROVED` | 농가 승인 완료 |
| `PROCUREMENT_PARTIAL_APPROVED` | 일부 수량 확인 필요 |
| `PROCUREMENT_REJECTED` | 농가 승인 불가 |
| `QUALITY_CHECKING` | 선별 중 |
| `READY_TO_SHIP` | 발송 준비 |
| `SHIPPED` | 배송 중 |
| `DELIVERED` | 배송 완료 |
| `RETURN_REQUESTED` | 반품 요청 접수 |
| `REFUNDED` | 환불 완료 |

프론트 구현 기준:

- 주문 상태는 `OrderStatusMapper.labelOf(statusCode)`에서 변환한다.
- 기획서의 `orders.order_status` 기준 상태값과 1차 대조 완료했다.
- 알 수 없는 상태값은 고객 화면에서 `상태 확인 중`으로 표시한다.

주문 상세 화면의 상태 타임라인 표시를 위해 아래 시간이 있으면 좋다.

```text
paid_at
procurement_approved_at
quality_checked_at
shipped_at
delivered_at
return_requested_at
```

## 4. 배송 정보 표시

배송 중 또는 배송 완료 상태에서는 택배사와 송장번호를 표시한다.

권장 API:

```text
GET /api/v1/me/orders/{order_id}/shipment
```

필요 응답 필드:

```text
carrier_name
tracking_number
shipment_status
shipped_at
delivered_at
```

프론트 표시 기준:

- `SHIPPED`, `DELIVERED`, `RETURN_REQUESTED` 상태에서는 택배사와 송장번호를 표시한다.
- 아직 배송 전이면 `배송 준비 중`, `송장번호 준비 중` 같은 고객 문구로 표시한다.
- 배송 상태는 `OrderStatusMapper.shipmentLabelOf(shipmentStatus)`에서 변환한다.

| 배송 상태 | 고객 표시 |
|---|---|
| `READY` | 배송 준비 |
| `SHIPPED` | 배송 중 |
| `DELIVERED` | 배송 완료 |

## 5. 반품 신청

현재 프론트 기준은 배송 완료 상태에서 반품 신청 버튼을 노출한다.

권장 API:

```text
POST /api/v1/returns
```

요청 필드 예시:

```json
{
  "order_id": 8,
  "reason_code": "DAMAGED",
  "reason_detail": "받은 사과 일부가 파손되어 반품을 요청합니다.",
  "request_amount": 39000,
  "evidence_image_urls": [
    "https://example.com/evidence-1.jpg"
  ]
}
```

프론트 검증 기준:

- 반품 상세 사유는 10자 이상 입력해야 한다.
- 부분 금액 선택은 제거하고 전체 반품 중심으로 정리했다.
- 반품 신청 후 주문 상태는 `RETURN_REQUESTED` 또는 고객 표시 `반품 요청 접수`로 보여준다.
- 반품 상태는 `OrderStatusMapper.returnLabelOf(returnStatus)`에서 변환한다.

| 반품 상태 | 고객 표시 |
|---|---|
| `REQUESTED` | 반품 요청 접수 |
| `APPROVED` | 반품 승인 |
| `REJECTED` | 반품 반려 |
| `REFUNDED` | 환불 완료 |

## 6. 결제

현재 결제 화면은 시연용 Mock 결제 기준이다.

권장 API:

```text
POST /api/v1/payments/mock-approve
```

요청 필드 예시:

```json
{
  "order_id": 8,
  "idempotency_key": "mock-pay-20260508-0001"
}
```

프론트 기준:

- `카드 간편결제`만 시연 결제로 진행한다.
- `무통장 입금`, `카카오페이`, `네이버페이`는 준비중 팝업을 표시한다.
- 실제 결제 연동 전까지 고객 화면에는 개발자용 `Mock` 표현을 직접 노출하지 않는다.

## 7. 카카오 주소 검색

카카오 주소 검색은 프론트에서 주소 검색 UI를 띄우고, 선택된 주소를 입력폼에 채우는 작업이다.

백엔드에서는 최종 저장값만 받으면 된다.

필요 저장 필드:

```text
zip_code
address
detail_address
```

적용 전 확인:

- Flutter Web 지원 패키지 또는 JS 연동 방식 검토가 필요하다.
- 프로젝트 규칙상 새 패키지는 pub.dev 확인 후 사용자 승인 뒤 추가한다.

## 8. 고객센터/푸터 정보

홈 하단 푸터에는 고객센터 전화번호와 이메일 문의 링크를 표시한다.

현재 프론트 더미 기준:

```text
전화번호: 010-8953-2362
이메일: support@harvestslot.kr
운영시간: 평일 AM 09:00 - PM 06:00
```

프론트 동작:

- 전화번호 클릭 시 `tel:` 링크로 전화 앱을 연다.
- 이메일 문의 클릭 시 `mailto:` 링크로 메일 앱을 연다.
- PC 브라우저처럼 전화 앱을 열 수 없는 환경에서는 고객용 안내 팝업을 표시한다.

백엔드 또는 관리자에서 내려줄지 결정할 항목:

```text
support_phone
support_email
support_hours
terms_url
privacy_url
```

MVP에서는 프론트 고정값으로도 가능하다. 다만 운영 정보가 자주 바뀔 가능성이 있으면 설정 API나 관리자 관리값으로 분리하는 것이 좋다.

## 9. 실제 API 연결 권장 순서

백엔드와 사용자 웹을 연결할 때는 아래 순서가 가장 안정적이다.

| 순서 | 연결 범위 | 주요 API | 이유 |
|---|---|---|---|
| 1 | 상품과 수확 슬롯 | `GET /products`, `GET /products/{product_id}`, `GET /products/{product_id}/slots` | 공개 데이터이며 홈, 목록, 상세, 예약 시작 흐름의 기준이 된다. |
| 2 | 인증과 로그인 세션 | `POST /auth/login`, `GET /me` | 예약, 주문, 마이페이지, 반품 API 호출 전 토큰 흐름이 먼저 필요하다. |
| 3 | 마이페이지와 배송지 | `GET /me`, `GET /me/addresses`, `POST /me/addresses` | 주문서 작성 화면의 배송지 선택과 자동 채움 검증에 필요하다. |
| 4 | 예약과 주문 생성 | `POST /reservations/preview`, `POST /reservations`, `POST /orders/from-reservation` | 수량, 가격, 슬롯 상태 검증은 서버가 최종 판단해야 한다. |
| 5 | 결제와 주문 조회 | `POST /payments/mock-approve`, `GET /me/orders`, `GET /me/orders/{order_id}` | 주문 상태, 배송 상태, 주문 상세 타임라인 확인에 필요하다. |
| 6 | 반품 | `POST /returns`, `GET /me/returns` | 배송 완료 주문 기준 반품 신청과 상태 변경을 연결한다. |
| 7 | 고객센터와 정책 링크 | 설정 API 또는 프론트 고정값 | MVP에서는 고정값으로 가능하며 운영 단계에서 분리할 수 있다. |

프론트 Repository 교체 기준:

- `ProductRepository`는 `ApiProductRepository`로 교체한다.
- `OrderRepository`는 `ApiOrderRepository`와 필요 시 `ApiReturnRepository`로 분리한다.
- `LocalBasketRepository`는 예약 생성 전 로컬 상태이므로 브라우저 저장소 기반으로 유지할 수 있다.
- 서버 상태 코드는 `OrderStatusMapper`로 고객용 한글 문구로 변환한다.
