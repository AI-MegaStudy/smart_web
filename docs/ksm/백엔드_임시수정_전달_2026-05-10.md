# 백엔드 임시수정 전달

작성일: 2026-05-10  
작성 목적: 사용자 웹 연동 중 백엔드와 실제 MySQL DB 스키마가 맞지 않아 임시 수정한 내용을 백엔드 담당자에게 전달하기 위함

## 0. 2026-05-10 최신 반영

팀장 전달 기준으로 이메일 인증 DB 문제는 해결되었다.

해결된 DB 항목:

- `email_verifications.attempt_count` 추가
- `email_verifications.updated_at` 추가
- 가입 전 인증을 위해 `email_verifications.account_id`를 `NULL` 허용
- 해시 저장을 위해 `email_verifications.verification_code`를 `VARCHAR(255)`로 변경

이에 따라 백엔드 임시 제거 내용은 복구 대상이다.

- `EmailVerification` 모델은 `TimestampMixin`을 다시 사용한다.
- 인증 코드 실패 시 `attempt_count`를 증가시키고 최대 실패 횟수를 제한한다.
- 가입 전 인증 후 회원가입이 완료되면 인증 레코드의 `account_id`를 새 계정과 연결한다.
- 환경변수는 `backend/.env`가 아니라 프로젝트 루트 `.env`를 기준으로 관리한다.

## 1. 수정 배경

사용자 웹에서 회원가입 API를 호출했습니다.

```http
POST /api/v1/auth/customers/signup
```

요청은 프론트에서 정상적으로 백엔드까지 전달됐지만, 백엔드에서 `500 Internal Server Error`가 발생했습니다.

처음 확인된 오류:

```text
Unknown column 'email_verifications.attempt_count' in 'field list'
```

이후 `attempt_count`를 제거한 뒤 다시 확인하니 다음 오류가 추가로 확인됐습니다.

```text
Unknown column 'email_verifications.updated_at' in 'field list'
```

즉 문제 원인은 프론트 요청값이 아니라 아래 불일치입니다.

```text
백엔드 SQLAlchemy 모델과 실제 MySQL DB의 email_verifications 테이블 구조가 다름
```

## 4. 수정 내용

### 4.1 `backend/app/models/account.py`

`EmailVerification` 모델에서 실제 DB에 없는 `attempt_count` 필드를 제거했습니다.

또한 `EmailVerification` 모델이 `TimestampMixin`을 상속하면 `created_at`, `updated_at`을 둘 다 자동 매핑합니다.

실제 DB에는 `created_at`은 있지만 `updated_at`은 없어서, `EmailVerification`만 `TimestampMixin` 상속을 제거하고 `created_at` 컬럼만 직접 선언했습니다.

변경 전 개념:

```python
class EmailVerification(Base, TimestampMixin):
    ...
    attempt_count: Mapped[int] = mapped_column(Integer, nullable=False, default=0)
```

변경 후 개념:

```python
class EmailVerification(Base):
    ...
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, nullable=False)
```

### 4.2 `backend/app/services/email_verification_service.py`

이메일 인증 생성과 검증 로직에서 `attempt_count` 사용 부분을 제거했습니다.

제거한 로직:

```python
attempt_count=0
```

```python
if verification.attempt_count >= settings.email_verification_max_attempts:
    raise HTTPException(status_code=429, detail="verification attempts exceeded")
```

```python
verification.attempt_count += 1
self.session.commit()
```

현재 임시 동작:

- 인증 코드가 틀리면 기존처럼 `invalid verification code` 오류를 반환합니다.
- 단, 인증 코드 입력 실패 횟수 제한 기능은 임시로 빠진 상태입니다.

## 6. 백엔드 담당자 확인 요청

최종 방향 결정이 필요합니다.

### 선택지 A. 현재 DB 스키마 유지

현재 실제 DB 기준에 맞춰 아래 컬럼 없이 운영합니다.

```text
attempt_count
updated_at
```

이 경우:

- 현재 임시 수정 방향 유지
- 이메일 인증 실패 횟수 제한 기능 없음
- `EmailVerification` 모델은 `TimestampMixin`을 상속하지 않고 `created_at`만 직접 사용

### 선택지 B. 백엔드 모델 기준으로 DB/ERD 수정

`attempt_count`, `updated_at`을 정식 기능으로 유지하려면 실제 DB와 ERD에 컬럼을 추가해야 합니다.

필요 SQL 예시:

```sql
ALTER TABLE email_verifications
ADD COLUMN attempt_count INT NOT NULL DEFAULT 0 AFTER verified_at;

ALTER TABLE email_verifications
ADD COLUMN updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
ON UPDATE CURRENT_TIMESTAMP AFTER created_at;
```

이 경우:

- ERD에도 `attempt_count:int`, `updated_at:datetime` 추가 필요
- 백엔드 모델에서 `TimestampMixin` 상속 복구 가능
- 이메일 인증 실패 횟수 제한 로직 복구 가능

## 7. 전달 요약

사용자 웹 회원가입 연동을 계속 진행하기 위해, 실제 DB에 없는 `attempt_count`, `updated_at` 참조를 백엔드 코드에서 임시 제거했습니다.

다만 이는 프론트 연동을 위한 임시 조치입니다. 백엔드 담당자는 실제 DB 스키마를 기준으로 유지할지, 아니면 백엔드 모델 기준으로 DB/ERD를 확장할지 결정해 주세요.

## 8. 2026-05-10 추가 확인: 가입 전 이메일 인증 DB 보정 완료

회원가입 전에 이메일 인증을 먼저 진행하는 흐름에서 아래 문제가 확인됐고, 이후 DB 보정으로 해결했습니다.

해결된 지점:

- 가입 전 인증 흐름에서는 아직 `accounts` 레코드가 없으므로 `email_verifications.account_id`는 `NULL`을 허용합니다.
- `attempt_count`, `updated_at` 컬럼을 DB에 추가했습니다.
- `verification_code`는 해시 저장을 위해 `VARCHAR(255)`로 변경했습니다.
- 실제 SMTP와 DB 설정은 프로젝트 루트 `.env` 기준으로 관리합니다.

백엔드 코드 반영 기준:

1. `EmailVerification` 모델에서 `TimestampMixin`을 다시 사용합니다.
2. 인증 코드 실패 시 `attempt_count`를 증가시키고 최대 실패 횟수를 제한합니다.
3. 가입 전 인증 후 회원가입이 완료되면 인증 레코드의 `account_id`를 새 계정과 연결합니다.
4. 인증 메일 발송 실패 시 프론트에서는 고객용 문구로 표시합니다.

현재 상태:

- 이메일 인증 기능은 DB 보정 완료 기준으로 재검증합니다.
- 고객 화면의 테스트 인증 코드 노출 문구는 제거했습니다.
- 회원가입/로그인 자체 QA는 새 계정 기준으로 다시 확인해야 합니다.

## 9. 백엔드 폴더 수정/추가 파일 전체 정리

2026-05-10 기준 `backend` 폴더에서 수정하거나 추가한 파일은 아래와 같습니다.

### 9.1 이메일 인증 DB 스키마 불일치 임시 수정

수정 파일:

```text
backend/app/models/account.py
backend/app/services/email_verification_service.py
```

수정 내용:

- `EmailVerification` 모델에서 실제 DB에 없는 `attempt_count` 필드 참조를 제거했습니다.
- `EmailVerification` 모델에서 `TimestampMixin` 상속을 제거했습니다.
- 실제 DB에는 `created_at`은 있지만 `updated_at`이 없어서, `created_at`만 직접 선언했습니다.
- 이메일 인증 코드 검증 로직에서 `attempt_count` 증가와 최대 실패 횟수 제한 로직을 제거했습니다.

주의 사항:

- 이 수정은 실제 DB 스키마에 맞춰 프론트 연동을 계속하기 위한 임시 조치입니다.
- 백엔드 담당자가 `attempt_count`, `updated_at`을 정식으로 사용할 계획이라면 DB와 ERD에 해당 컬럼을 추가해야 합니다.
- 현재는 인증 코드 실패 횟수 제한 기능이 빠져 있습니다.

### 9.2 회원정보 수정 API 추가

수정 파일:

```text
backend/app/schemas/auth_schema.py
backend/app/services/auth_service.py
backend/app/routers/auth_router.py
```

추가 API:

```text
PUT /api/v1/me
```

추가 목적:

- 사용자 웹의 회원정보 수정 화면에서 이름과 전화번호를 저장하기 위해 추가했습니다.
- 저장 후 `GET /api/v1/me`와 같은 형태의 최신 회원 정보를 다시 반환하도록 했습니다.

요청 필드:

```text
name

```

처리 방식:

- 고객 계정이면 `customer_profiles.customer_name`, `customer_profiles.customer_phone`을 수정합니다.
- 농가 계정이면 `owner_profiles.owner_name`, `owner_profiles.owner_phone`을 수정합니다.
- 프로필이 없으면 `profile not found` 오류를 반환합니다.

프론트 확인 상태:

- 회원정보 수정 후 새로고침 없이 회원정보 보기 화면에 변경값이 반영되는 것을 확인했습니다.

## 10. 백엔드 담당자에게 전달할 핵심 요약

전달 요약:

1. 실제 DB의 `email_verifications`에는 `attempt_count`, `updated_at`이 없어서 백엔드 모델과 불일치했습니다.
2. 프론트 연동을 계속하기 위해 두 필드 참조를 임시 제거했습니다.
3. 가입 전 이메일 인증을 위해 `email_verifications.account_id`는 `NULL`을 허용합니다.
4. SMTP 설정은 실제 FastAPI가 읽는 프로젝트 루트 `.env` 기준으로 확정합니다.
5. 사용자 웹 구현을 위해 `PUT /api/v1/me`와 `/api/v1/me/addresses` 계열 API를 추가했습니다.
6. 배송지 API는 별도 배송지 테이블 없이 `customer_profiles` 기본 배송지 필드를 사용하는 MVP 구현입니다.
7. 여러 배송지 정식 관리가 필요하면 별도 배송지 테이블과 API 재설계가 필요합니다.
