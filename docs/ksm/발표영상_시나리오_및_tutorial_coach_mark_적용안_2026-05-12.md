# SMART_WEB 발표영상 시나리오 및 `tutorial_coach_mark` 적용안

작성일: 2026-05-12

## 1. 문서 목적

이 문서는 팀장이 `SMART_WEB` 전체 흐름을 빠르게 이해하고, 발표용 영상과 발표 대본을 만들 수 있도록 정리한 시나리오 문서다.

추가로 이후 `tutorial_coach_mark: ^1.3.3` 패키지를 사용해 화면 안내형 데모를 만들 때 기준이 되는 화면 순서와 강조 포인트도 함께 정리한다.

## 2. 발표영상 목표

발표영상에서 가장 먼저 전달해야 하는 메시지는 아래와 같다.

- 고객이 상품을 고르고
- 수확 슬롯을 예약하고
- 주문과 결제를 진행하고
- 배송과 반품까지 확인할 수 있는
- 사용자 예약 웹을 Flutter Web으로 구현했다

즉, 기술 설명보다 먼저 `처음 방문한 고객이 실제로 예약을 완료하는 흐름`이 자연스럽게 보이도록 구성하는 것이 중요하다.

## 3. 발표영상 기본 방향

- 개발자용 용어보다 고객 입장 표현을 우선 사용한다.
- `Mock`, `QA`, `DB`, `API`, `테스트`, 내부 상태코드 같은 표현은 발표 화면 문구로 직접 노출하지 않는다.
- 영상은 기능 나열보다 하나의 고객 여정을 따라가는 방식으로 구성한다.
- 팀장 발표 멘트는 짧고 명확하게 유지한다.
- 화면 전환마다 "왜 이 화면이 필요한지"가 드러나야 한다.

## 4. 권장 영상 길이

- 짧은 버전: 2분 30초 ~ 3분 30초
- 발표용 표준 버전: 4분 ~ 5분 30초

현재 기준으로는 발표용 표준 버전이 가장 적합하다.

## 5. 발표 시나리오 요약

발표 시나리오는 아래 두 가지 흐름으로 나누어 이해하면 좋다.

### 5.1 전체 서비스 흐름

처음 방문한 고객 기준 전체 흐름은 아래 순서가 맞다.

```text
회원가입
-> 이메일 인증
-> 로그인
-> 홈
-> 홈 섹션 빠른 이동 메뉴 확인
-> 상품 목록
-> 상품 상세
-> 수확 슬롯 선택
-> 예약함
-> 예약 확인
-> 주문서 작성
-> 결제
-> 주문 완료
-> 주문 상세
-> 배송 확인
-> 반품 신청
```

### 5.2 발표용 핵심 시연 흐름

다만 발표 영상에서는 시간상 회원가입과 인증 과정을 모두 길게 담기보다,
로그인 완료 상태에서 핵심 구매 흐름을 보여주는 방식이 더 깔끔할 수 있다.

```text
홈
-> 홈 섹션 빠른 이동 메뉴 확인
-> 상품 목록
-> 상품 상세
-> 수확 슬롯 선택
-> 예약함
-> 예약 확인
-> 주문서 작성
-> 결제
-> 주문 완료
-> 주문 상세
-> 배송 확인
-> 반품 신청
```

## 6. 발표영상 상세 시나리오

### 6.1 오프닝

목적:

- 서비스가 무엇인지 한 문장으로 이해시키는 구간

권장 멘트:

```text
SMART_WEB은 고객이 농가의 수확 일정에 맞춰 신선한 과일을 예약하고,
주문과 배송, 반품까지 한 번에 확인할 수 있는 Flutter Web 사용자 예약 서비스입니다.
```

화면 구성:

- 홈 첫 화면 진입
- 브랜드 인트로 또는 대표 비주얼
- 대표 상품 진입 버튼 또는 메인 CTA 노출

### 6.2 회원가입, 이메일 인증, 로그인

목적:

- 고객이 실제로는 인증 기반으로 서비스를 시작한다는 점 설명

권장 멘트:

```text
처음 방문한 고객은 회원가입과 이메일 인증을 거쳐 로그인한 뒤
예약과 주문 기능을 이용할 수 있습니다.
```

영상 운영 팁:

- 전체 흐름 설명용 영상이면 이 구간을 짧게 포함한다.
- 핵심 구매 시연 영상이면 이 구간은 짧게 스킵하거나,
  "로그인 후부터 시연하겠습니다"라는 멘트로 전환해도 된다.

보여줄 요소:

- 회원가입 화면
- 이메일 인증 화면
- 로그인 화면

### 6.3 홈과 섹션 빠른 이동 메뉴

목적:

- 홈 화면이 단순 첫 화면이 아니라 주요 소개 섹션으로 빠르게 이동할 수 있는 구조임을 보여주기

권장 멘트:

```text
로그인 후 홈 화면에서는 주요 소개 섹션을 바로 이동할 수 있는 빠른 메뉴를 통해
상품, 품종, 보관, 농촌, FAQ 정보를 쉽게 탐색할 수 있습니다.
```

보여줄 요소:

- 홈 메인 비주얼
- 우측 플로팅 빠른 이동 메뉴
- `위로`
- `상품`
- `품종`
- `보관`
- `농촌`
- `FAQ`

발표 포인트:

- 이 메뉴는 발표자가 홈의 주요 섹션을 빠르게 보여줄 때도 유용하다.
- 홈 소개형 콘텐츠와 실제 상품 탐색 흐름을 자연스럽게 연결해준다.

### 6.4 상품 탐색

목적:

- 고객이 예약 가능한 상품을 빠르게 찾는 흐름 소개

권장 멘트:

```text
고객은 상품 목록에서 현재 예약 가능한 상품과 수확 예정 범위,
가격과 잔여 수량 정보를 한눈에 확인할 수 있습니다.
```

보여줄 요소:

- 상품 카드
- 품종
- 가격
- 수확 예정 범위
- 잔여 수량 또는 예약 가능 상태

### 6.5 상품 상세와 농가 정보

목적:

- 단순 상품 판매가 아니라 농가 기반 예약 서비스라는 점 강조

권장 멘트:

```text
상품 상세 화면에서는 상품 설명뿐 아니라 농가 정보,
배송 안내, 반품 안내까지 함께 확인할 수 있습니다.
```

보여줄 요소:

- 상품 대표 이미지
- 농가명, 지역
- 상품 설명
- 배송 안내
- 반품 안내

### 6.6 수확 슬롯 선택

목적:

- 이 서비스의 핵심 차별점인 수확 예약 흐름 강조

권장 멘트:

```text
고객은 일반 재고 구매가 아니라,
농가가 확정한 수확 슬롯을 기준으로 예약을 진행합니다.
```

추가 멘트:

```text
박스 단위와 수확 예정 범위를 선택하면,
예약 가능한 수량과 금액이 즉시 반영됩니다.
```

보여줄 요소:

- 패키지 단위 선택
- 수확 슬롯 카드
- 예약 가능 수량
- 박스 수량 선택
- 예약 요약

### 6.7 예약함

목적:

- 서버 예약 전 단계인 로컬 예약함 개념 전달

권장 멘트:

```text
선택한 상품은 우선 예약함에 담기며,
고객은 여러 예약 항목을 검토한 뒤 다음 단계로 이동할 수 있습니다.
```

보여줄 요소:

- 선택된 상품 목록
- 박스 수량
- 총 중량
- 총 금액

### 6.8 예약 확인

목적:

- 실제 예약 생성 직전 검토 단계 설명

권장 멘트:

```text
예약 확인 단계에서는 수확 일정과 수량, 금액을 다시 확인하고
안내 문구를 통해 일정 변동 가능성도 함께 전달합니다.
```

강조 문구:

```text
농가가 확정한 수확 예정 범위입니다.
기상과 생육 상황에 따라 일정이 조정될 수 있습니다.
```

### 6.9 주문서 작성

목적:

- 예약에서 주문으로 넘어가는 흐름 설명

권장 멘트:

```text
예약이 완료되면 고객은 수령인 정보와 배송지를 입력해
주문서를 작성할 수 있습니다.
```

보여줄 요소:

- 수령인 이름
- 연락처
- 배송지
- 배송 요청사항

### 6.10 결제

목적:

- 최종 결제 금액 확인과 결제 완료 흐름 전달

권장 멘트:

```text
결제 단계에서는 상품 금액과 최종 결제 금액을 확인한 뒤
결제를 완료할 수 있습니다.
```

보여줄 요소:

- 상품 금액
- 배송 정보 요약
- 최종 결제 금액
- 결제 동의 또는 결제 버튼

### 6.11 주문 완료

목적:

- 주문이 정상적으로 생성됐음을 보여주는 안심 구간

권장 멘트:

```text
결제가 끝나면 주문 번호와 상태를 즉시 확인할 수 있고,
이후 진행 상황은 주문 상세에서 계속 추적할 수 있습니다.
```

보여줄 요소:

- 주문 번호
- 결제 금액
- 현재 주문 상태

### 6.12 주문 상세와 배송 확인

목적:

- 고객이 주문 이후에도 상태를 계속 확인할 수 있음을 보여주기

권장 멘트:

```text
주문 상세에서는 결제 이후 농가 확인, 배송 진행,
배송 완료까지의 상태를 한 화면에서 확인할 수 있습니다.
```

보여줄 요소:

- 주문 상태
- 상품 정보
- 배송 정보
- 상태 타임라인

### 6.13 반품 신청

목적:

- 구매 이후 사후 처리까지 지원한다는 점 전달

권장 멘트:

```text
배송 완료 이후에는 반품 신청도 웹에서 직접 진행할 수 있습니다.
고객은 반품 사유와 상세 사유, 증빙 이미지를 첨부해 요청할 수 있습니다.
```

보여줄 요소:

- 반품 사유 선택
- 상세 입력
- 증빙 이미지 첨부
- 반품 신청 버튼

### 6.14 마무리

목적:

- 전체 서비스 가치를 다시 한 문장으로 회수

권장 멘트:

```text
정리하면 SMART_WEB은 상품 탐색부터 수확 예약,
주문, 결제, 배송, 반품까지 고객의 전체 예약 구매 여정을
하나의 웹 서비스에서 자연스럽게 연결한 사용자 예약 웹입니다.
```

## 7. 팀장 발표용 짧은 대본

아래 대본은 1차 발표 초안으로 바로 읽을 수 있는 형태다.

```text
지금 보시는 화면은 Harvest Slot 사용자 예약 웹입니다.
고객은 회원가입과 이메일 인증, 로그인을 거친 뒤
홈에서 주요 소개 섹션을 빠르게 둘러본 뒤
상품 목록에서 예약 가능한 과일 상품을 확인하고,
상품 상세에서 농가 정보와 배송, 반품 안내를 함께 볼 수 있습니다.

이 서비스의 핵심은 수확 슬롯 기반 예약입니다.
고객은 원하는 박스 단위와 수확 예정 범위를 선택해 예약을 진행하고,
예약함과 예약 확인 화면에서 수량과 금액을 검토할 수 있습니다.

이후 주문서 작성과 결제를 완료하면 주문 상세에서 진행 상태와 배송 현황을 확인할 수 있고,
배송 완료 후에는 반품 신청까지 웹에서 직접 처리할 수 있습니다.

즉, 상품 탐색부터 예약, 주문, 결제, 배송, 반품까지의 전체 흐름을
고객 중심으로 하나의 Flutter Web 서비스에 연결한 것이 이번 SMART_WEB의 핵심입니다.
```

## 8. 영상 촬영 순서 제안

### 8.1 추천 장면 순서

```text
1. 회원가입 또는 로그인 안내
2. 홈
3. 홈 빠른 이동 메뉴 확인
4. 상품 목록
5. 상품 상세
6. 수확 슬롯 선택
7. 예약함
8. 예약 확인
9. 주문서 작성
10. 결제
11. 주문 완료
12. 주문 상세
13. 배송 정보
14. 반품 신청
15. 마무리 화면
```

### 8.2 장면 전환 방식

- 첫 화면은 천천히 시작한다.
- 핵심 버튼 클릭은 멈칫하지 말고 자연스럽게 이어간다.
- 글을 읽어야 하는 화면은 2초 이상 머문다.
- 주문 상세와 반품 신청은 발표자 설명이 들어가므로 조금 더 길게 보여준다.

## 9. 촬영 전 준비 체크리스트

- 발표용 계정 로그인 상태 확인
- 대표 상품이 예약 가능한 상태인지 확인
- 상품 이미지와 농가 이미지 노출 확인
- 예약함이 비어 있는 상태에서 시작
- 주문 완료 후 조회 가능한 주문 데이터 준비
- 배송 완료 상태 주문과 반품 신청 가능한 주문 준비
- 고객 화면에 개발자용 문구가 없는지 최종 점검

## 10. `tutorial_coach_mark` 적용 목적

발표용 영상에서는 클릭 흐름만 보여줘도 되지만,
실제 시연용 데모에서는 화면 핵심 영역을 단계별로 안내하는 방식이 더 효과적일 수 있다.

`tutorial_coach_mark: ^1.3.3`는 아래 목적에 적합하다.

- 팀장 발표 리허설용 가이드
- 시연 중 주요 기능 포인트 자동 강조
- 처음 보는 사람에게 핵심 기능 설명

- 반품 신청

### 11.2 화면별 코치마크 포인트

- 로그인 버튼

처음 이용하는 고객은 회원가입과 인증 후 로그인해 예약을 시작할 수 있습니다.

#### 홈

- 대표 상품 진입 버튼
- 서비스 핵심 소개 영역

예시 문구:

```text
여기서 바로 대표 상품으로 이동해 예약을 시작할 수 있습니다.

- `농촌`
- `FAQ`

예시 문구:

홈 화면에서는 빠른 이동 메뉴를 통해 주요 소개 섹션을 바로 확인할 수 있습니다.
```

#### 상품 목록

- 상품 카드
- 가격
- 수확 예정 범위
- 예약 가능 상태

예시 문구:

```text
상품별 수확 예정 범위와 예약 가능 상태를 비교해 원하는 상품을 고를 수 있습니다.
```

#### 상품 상세

- 상품 이미지
- 농가 정보
- 배송 안내
- 반품 안내
- 수확 슬롯 영역

예시 문구:

```text
상품 설명과 농가 정보를 확인한 뒤, 아래에서 원하는 수확 슬롯을 선택합니다.

주문 이후 진행 상태와 배송 현황을 한 화면에서 확인할 수 있습니다.

배송 완료 주문에 대해서는 웹에서 직접 반품 요청을 접수할 수 있습니다.
```

## 12. `tutorial_coach_mark` 구현 준비 메모

구현할 때는 각 대상 위젯에 `GlobalKey`를 부여하고, 화면 진입 후 코치마크를 순차 실행하는 방식이 적합하다.

권장 방식:

```text
View
-> 주요 버튼/카드/요약 영역에 GlobalKey 부여
-> ViewModel 또는 화면 상태에서 첫 진입 여부 판단
-> tutorial_coach_mark 순차 실행
```

주의사항:

- 고객 실제 사용 흐름에서는 첫 진입 시 자동 실행 여부를 신중히 결정한다.
- 발표용 모드와 일반 사용자 모드를 분리하는 것이 좋다.
- 모바일과 웹에서 강조 위치가 달라질 수 있으므로 레이아웃별 확인이 필요하다.
