# 사용자 웹 작업 진행 기록

작성일: 2026-05-02

이 문서는 `SMART_WEB` 사용자 웹에서 지금까지 진행한 디자인 및 화면 구성 작업을 정리한 기록이다.

## 1. 작업 목표

`SMART_WEB`은 Harvest Slot 사용자 예약 웹을 Flutter Web으로 구현하는 프로젝트다.

이번 작업의 중심 목표는 아래와 같다.

- 사용자 웹 주요 예약 흐름 화면 구성
- 프로토타입 기준 디자인 반영
- 사용자 입장에서 자연스러운 화면 문구 정리
- 반응형 웹 화면 구조 정리
- 화면 상단 브랜드 AppBar 통일
- 추후 백엔드 연결 전 더미 데이터 기반 화면 흐름 완성

## 2. 참고한 주요 자료

## 3. 전체 구조

Flutter Web 기준으로 MVVM 구조를 유지했다.

```text
lib/
  app/
  core/
  data/
    models/
    repositories/
  presentation/
    view_models/
    views/
    widgets/
```

화면은 `presentation/views`, 화면 상태는 `presentation/view_models`, 더미 데이터와 모델은 `data` 아래에서 관리한다.

## 4. 구현 및 수정한 주요 화면

### 홈 화면

- 프로토타입의 고객 예약 흐름 전체 시안을 기준으로 재구성했다.
- 상단 브랜드, 상품, 내 주문, 회원가입, 로그인, 예약함 메뉴를 배치했다.
- 반응형에서 메뉴가 깨지지 않도록 AppBar 구조를 정리했다.
- 대표 상품 영역과 수확 예정 날짜 표시를 반영했다.
- `Harvest Slot` 브랜드 클릭 시 홈으로 이동하도록 했다.

### 상품 목록 화면

- 고객용 문구로 정리했다.
- 상품 목록 상단을 `Harvest Slot` 브랜드 링크 형태로 변경했다.
- 상품 카드에서 불필요한 `5kg 박스` 표시를 정리했다.
- 관리자용 느낌이 강한 문구를 고객용 안내 문구로 바꿨다.

### 상품 상세 화면

- `product-detail.html` 프로토타입을 기준으로 화면을 재구성했다.
- 상품 이미지, 상품 설명, 패키지 단위, 수확 슬롯 선택, 박스 수량 선택, 예약함 담기 흐름을 구성했다.
- 왼쪽 빈 영역에는 상품 설명과 수확 직배송 안내 카드 성격의 콘텐츠를 추가했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 담아둔 수확 상품 화면

- 로컬 예약함 화면을 구성했다.
- 담긴 상품을 확인하고, 예약 전 확인으로 넘어갈 수 있게 했다.
- 담긴 상품을 제거할 수 있는 버튼과 안내 문구를 추가했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 예약 전 확인 화면

- 예약 생성 전 상품, 수량, 가격을 확인하는 화면으로 구성했다.
- 고객이 실제 이용하는 흐름에 맞게 문구를 정리했다.
- 수확 일정 변동 가능성 안내를 유지했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 주문서 작성 화면

- `checkout.html` 프로토타입을 기준으로 디자인을 재구성했다.
- 배송지, 배송 메모, 주문 요약, 결제 진행 버튼 영역을 구성했다.
- 기본 배송지 선택처럼 보이는 카드 UI를 반영했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 결제 화면

- 고객 화면에 `Mock 결제`처럼 개발용 문구가 보이지 않도록 정리했다.
- 결제 대기, 결제 금액, 결제 완료 흐름이 자연스럽게 보이도록 수정했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 주문 완료 화면

- 주문 완료 후 고객에게 보여줄 완료 카드 화면을 구성했다.
- 주문번호, 결제 금액, 농가 확인 중 상태를 안내한다.
- 홈으로 이동, 주문 상세 보기 버튼을 제공한다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 내 주문 화면

- 주문 목록과 주문 상태를 확인하는 화면이다.
- 주문 상세로 이동할 수 있게 유지했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 주문 상세 화면

- 주문 진행 상태 영역을 왼쪽 기준으로 정렬했다.
- 상태 타임라인을 고객이 보기 쉬운 `진행 현황` 형태로 변경했다.
- 중복으로 보이던 배송사 문구를 자연스러운 배송 진행 문구로 바꿨다.
- 반품 신청 버튼과 주문 요약 영역을 유지했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 반품 신청 화면

- `return-request.html` 프로토타입을 기준으로 다시 구성했다.
- 반품할 상품 선택, 반품 사유 카드, 상세 사유, 증빙 이미지, 요청 금액, 반품 요청 요약을 구성했다.
- 고객이 실제 이용하는 문구로 전체 텍스트를 정리했다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 반품 요청 완료 화면

- 반품 요청 후 바로 홈으로 이동하지 않고 확인 페이지로 이동하도록 새 화면을 추가했다.
- 반품 요청 접수 완료, 주문번호, 농가 확인 후 안내 상태를 보여준다.
- 홈으로 이동, 내 주문 보기 버튼을 제공한다.
- 상단 AppBar를 브랜드 AppBar로 통일했다.

### 로그인, 회원가입, 이메일 인증 화면

- `login.html`, `signup.html`, `email-verify.html` 프로토타입을 기준으로 다시 구성했다.
- 왼쪽은 이미지 패널, 오른쪽은 흰색 카드 폼으로 구성했다.
- 외부 이미지 로딩 실패 시 초록 그라데이션 fallback이 보이도록 했다.
- 로그인, 회원가입, 이메일 인증 화면 모두 상단 AppBar를 `Harvest Slot` 브랜드 링크로 변경했다.
- 이메일 인증 코드 박스는 한 줄로 보이도록 크기와 배치를 조정했다.

## 2026-05-09 백엔드 상품 API 연결 작업 기록

백엔드 코드를 `SMART_WEB/backend` 안에서 확인하고, 사용자 웹의 상품/수확 슬롯 영역부터 실제 API 연결을 시작했다.

### 백엔드 실행 및 환경 확인

- 백엔드 구조는 FastAPI + MySQL 기준으로 확인했다.
- 실행 진입점은 `backend.app.main:app` 기준이다.
- API 기본 경로는 `/api/v1` 기준이다.
- 프론트 개발 기본 Base URL은 `http://localhost:8000/api/v1` 또는 `http://127.0.0.1:8000/api/v1`로 둔다.
- `backend/.env`를 읽을 수 있도록 백엔드 설정 파일을 보완했다.
- 로컬 확인 기준으로 health API와 DB health API가 정상 응답하는 것을 확인했다.

### 프론트 API 계층 추가

아래 파일을 추가했다.

```text
lib/core/api/api_client.dart
lib/core/api/api_exception.dart
lib/data/repositories/product_api_repository.dart
```

적용 기준:

- View에서 API를 직접 호출하지 않는다.
- ViewModel은 기존 Repository Contract에 의존한다.
- 실제 HTTP 통신은 Repository와 ApiClient가 담당한다.
- 백엔드 오류 메시지는 고객용 한글 메시지로 변환한다.

### 상품/슬롯 API 연결

연결한 API:

```http
GET /api/v1/products?featured=true
GET /api/v1/products
GET /api/v1/products/{product_id}
GET /api/v1/products/{product_id}/slots
```

반영한 화면:

- 홈 대표 상품
- 상품 목록
- 상품 상세
- 예약 확인에서 상품 정보 참조

표시 기준:

- 상품명은 백엔드 `product_name` 원문을 그대로 표시한다.
- 수확 예정일, 잔여 수량, 가격은 상품 응답보다 슬롯 조회 결과를 우선 사용한다.
- 상품 상세 예약 가능 여부는 실제 슬롯 조회 결과를 기준으로 판단한다.
- 슬롯이 없거나 잔여 수량이 없는 상품은 `수확 일정 준비중` 안내 상태로 표시한다.

### 발견한 이슈와 정리

- 상품 목록에서는 슬롯을 따로 조회해서 예약 가능 상품처럼 보였지만, 상품 상세에서는 상세 응답의 `open_slot_count`만 보고 예약 불가처럼 보이는 문제가 있었다.
- 이를 수정해 상세 화면도 슬롯 조회 결과가 있으면 예약 가능 화면으로 표시하도록 정리했다.
- 백엔드 더미 상품명에 `예약 사과 5kg`처럼 들어오는 값은 프론트에서 임의로 줄이지 않고 원문을 표시한다.
- 고객 화면에서 최종적으로 원하는 상품명은 백엔드 더미 데이터 또는 운영 데이터 입력 기준으로 맞추는 것이 좋다.

### 검증 상태

- `flutter analyze` 결과 `No issues found!`를 확인했다.
- `dart format`은 Windows 환경에서 Dart/Flutter 분석 서버 프로세스가 남아 오래 걸리는 현상이 있었다.
